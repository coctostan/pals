# v2.41 CODI v0.1 — Cross-Repo Trial on quark

**One-line premise:** pals is a bad CODI trial environment (1 TS file, 1,331 lines, markdown-dominated). quark is a good one (36k TS, 111 phases of PALS history, pre-indexed `.codegraph/graph.db`). The question Phase 177 could not answer — *does CODI's blast_radius materially shape APPLY when the phase's own work actually touches indexed TS?* — can be answered in quark. **Trial fuel for Phase 179: the user's in-flight quark patches, authored before CODI was enabled — causally independent of CODI, zero selection bias.**

---

## Why this milestone, why now

Phase 177 closed v2.40 with verdict `ITERATE_V0_1` and `S_pass_nonnull = 3`, mechanically crossing the SHIP threshold for the first time but failing the SHIP gate on `Q2=uncertain`. Under the unchanged Phase 173 rubric, Q2 requires ≥ 2 of 4 signals at projected-pass; only Signal 3 reaches it.

The blocker on Q2 is **structural**, not design-iterative: v2.40 produced exactly two non-CODI_NULL live rows, and in both cases APPLY's work did not consume the injected call-sites. That's not CODI failing — it's pals not having any indexed TS scope for APPLY to consume. Signal 4 is permanently stuck at `not-applicable` for the same reason.

Three more iterations inside pals will not resolve Q2. The thesis needs a test bed with real TS work.

## Why quark specifically

Scouted 2026-04-18 against 20+ candidate projects.

| Criterion | quark | Closest runner-up (wealthVizr) |
|-----------|-------|-------------------------------|
| Indexed TS (src/) | ~15.6k lines non-test, 36k total | ~104k lines |
| `.codegraph/graph.db` pre-indexed | **Yes (1.7MB, Mar 28)** | No — would need indexing |
| PALS already installed | **Yes (111 phases)** | No — brownfield init required |
| Recent phases with natural TS scope | Yes (110: 13 files; 111: 7 files in `src/kernel/`) | Unknown |
| Test suite exists | 57 test files, 950 tests | Unknown |
| Natural fan-out structure | `src/kernel/{loop,state,capabilities,process-syscalls}.ts` — exactly the call-graph shape Session D projected | TBD |
| Added friction to start trial | CODI install + `modules.yaml` regen | Brownfield init + index codebase + CODI install |

quark's Phase 110 is a near-perfect CODI test case: 13 files in `src/kernel/` with explicit call-site relationships (`vfs-resolve.ts`, `capabilities.ts`, `process-syscalls.ts`, `loop.test.ts`). Running the v2.40 extractor against a plan of that shape would produce a non-trivial Blast Radius, and APPLY would actually traverse those call-sites.

## What this milestone is NOT

- Not a new feature for CODI. Zero CODI-source code changes planned in pals.
- Not a rubric change. The Phase 173 / 177 decision rule stays unchanged.
- Not a cross-repo integration effort. quark stays a separate repo; no shared state.
- Not a migration of PALS itself to quark. quark already uses PALS.

## What this milestone IS

A bounded observation of CODI running against real TS work in a separate repo — specifically the user's in-flight quark patches applied under CODI as one or more natural quark PALS phases — evaluated against the unchanged Phase 173/177 rubric, to resolve the Q2 question definitively. Patches authored before CODI was enabled guarantee scope is not tuned to flatter the module.

## Proposed phase shape (v2.41, 3 phases)

### Phase 178 — CODI installation & baseline in quark

**Type:** setup
**Scope:** quark repo
**Deliverables:**
1. `pals.json` in quark gets a `modules.codi` entry (default-on, same shape as pals's entry).
2. quark's `modules.yaml` regenerated (via `bash drivers/pi/install.sh` from pals, or quark's equivalent install path) so the installed CODI pre-plan + post-unify hooks land.
3. `.paul/CODI-HISTORY.md` created (header + 0 rows — hook will populate organically).
4. Trial-integrity precheck: installed CODI manifest parity with pals's repo-source CODI manifest confirmed (Milestone Invariant 3 equivalent).
5. Baseline record: quark currently has 111 completed phases, none observed under CODI. This phase adds no rows; it sets up the instrument.
6. Validation: `npx vitest run` passes (quark's current 950-test baseline).

**Trial integrity constraint:** quark's `modules/codi/module.yaml` must exactly match pals's repo-source at the hash recorded at Phase 178 start. If pals's CODI source drifts during v2.41, future phases exclude themselves from the trial set until parity is restored.

**Exit criterion:** a synthetic dry-run plan in quark (or quark's next natural plan, whichever comes first) dispatches CODI end-to-end, appends one row to `.paul/CODI-HISTORY.md`, and the quality surface stays green.

### Phase 179 — Organic observation window (patch-driven)

**Type:** observation (no CODI-source edits in pals; no new code in quark beyond what quark's in-flight patch work produces)
**Scope:** the user's in-flight quark patches — executed under CODI as one or more natural quark PALS phases
**Trial fuel:** patches were authored WITHOUT CODI in mind, so scope selection is causally independent of CODI. This is strictly better than observing arbitrary future natural work, because selection bias is zero — the patches are what they are.
**Deliverables:**
1. Plan and execute the quark patch work as one or more normal quark PALS phases under CODI.
2. Each phase's PLAN runs the now-installed CODI pre-plan hook organically.
3. Each phase's UNIFY appends a row to quark's `.paul/CODI-HISTORY.md`.
4. No modifications to the pre-plan or post-unify hook contracts. No modifications to the extractor. Pure observation.
5. For each phase, record: did APPLY actually consume the injected blast_radius in its `<boundaries>` or task actions? Was there a call-site in the Blast Radius that mattered to the work? This is the qualitative data Phase 177 couldn't collect.
6. Zero controlled self-trials. Patches represent engineering intent that existed before CODI was enabled; they are not constructed to flatter the module.

**Naturality discipline (REQUIRED):** When planning each quark phase that applies these patches, draft `<context>` source files and `files_modified` lists from quark's engineering logic first, BEFORE CODI dispatch. Do not tune scope to feed CODI nicer symbols. CODI must see whatever the unbiased planner would have written anyway; if that produces `skipped-no-symbols` rows, those are honest data. The whole point is to escape Session C-style self-fulfilling evidence.

**Trial integrity:** every session's trial-integrity check re-confirms pals→quark parity on installed CODI (shared registry at `~/.pi/agent/skills/pals/modules.yaml`; SHA pinned in Phase 178 SUMMARY). Sessions against a stale installed manifest excluded.

**Exit criterion:** all patch work has landed in quark as one or more PALS phases. Time-based windows do not apply — the patches define the observation scope. If patches span 1 phase, Phase 180 proceeds with 1 row plus whatever counterfactual evidence it brings forward; if they span 3+ phases, more rows. Row count scales with engineering reality, not trial convenience.

- No retroactive CODI runs against quark's 111 completed phases. Only phases executed *under* installed CODI count as live evidence.
- No cross-repo state sharing beyond `.paul/CODI-HISTORY.md` rows being copied into pals's Phase 180 TRIAL-DATA as evidence.
- No expansion of the extractor to cover non-TS/JS surfaces during v2.41 — the whole point is to test the current extractor against natural TS scope.
- No Session C-style controlled self-trials. Phase 179 rejects controlled evidence entirely. If only 1–2 natural rows accumulate in the observation window, that is itself informative (and points toward KILL rather than toward constructing synthetic data).

## Integrity invariants (carried forward from v2.40, adapted)

The in-flight quark patches satisfy this by construction — they were authored before CODI was enabled in quark, so their scope is causally independent of CODI.
