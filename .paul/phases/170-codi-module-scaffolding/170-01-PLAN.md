<objective>

## Goal
Scaffold the CODI PALS module as a registry-discovered `pre-plan` advisory overlay that, on dispatch, unconditionally emits a single hello-world log line. No codegraph tools are called and no prerequisite checks are performed at this stage — the sole purpose is to prove the dispatch plumbing end-to-end before Phase 171 adds `impact` integration with call-time failure handling.

## Purpose
Phase 170 validates that the installed module registry, installer, and shared dispatch machinery can carry a new first-party module without any kernel or workflow changes. Establishing this baseline lets Phase 171 focus purely on structural-truth content (symbol extraction + `impact` injection) and prevents the common failure mode where a new module's plumbing is conflated with its logic. It also honors the integration plan's explicit "first concrete step" discipline: *"a hook implementation that does nothing but log `[dispatch] CODI: hello, objective=...` ... confirm the dispatch log appears in the plan. That validates the plumbing before any codegraph integration exists."*

## Output
- `modules/codi/module.yaml` — module manifest declaring one `pre-plan` advisory hook at priority 220 with a hello-world description (no skip paths, no tool calls).
- `modules/codi/references/codi.md` — reference doc explaining CODI's v0.1 hello-world scope, call-time failure-handling philosophy, codegraph tool dependency, and the deferred v0.2 → v0.5+ roadmap.
- Re-generated installed registry at `~/.pi/agent/skills/pals/modules.yaml` containing a `codi:` entry with matching `hook_details.pre-plan` metadata — produced by running the existing driver installer; no installer code changes.

</objective>
