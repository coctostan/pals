    Create a practical, step-by-step guide for building a new PALS module.
    Structure:
    1. Overview — what a module is, how it fits the architecture
    2. Prerequisites — what you need before starting
    3. Step-by-step walkthrough:
       a. Create module directory under modules/{name}/
       b. Write module.yaml manifest (identity, dependencies, refs, hooks)
       c. Create reference files (with 150-line budget per file)

       e. Add kernel command (optional) — workflow in kernel/commands/
       f. Wire into installer — what install.sh needs
       g. Add to pals.json schema
    4. Example: walk through an existing module (DEAN is simplest — pre-plan + post-apply, 2 refs)
    5. Checklist — everything to verify before shipping

    Source from: docs/MODULE-SPEC.md (formal spec), existing module.yaml files (real examples),
    kernel/workflows/ (hook dispatch patterns), install.sh (installer wiring).

    Tone: practical and direct. Not a formal spec (MODULE-SPEC.md already is that).
    Avoid: duplicating the full MODULE-SPEC.md content — reference it instead.
  </action>

<task type="auto">
  <name>Task 2: Create Troubleshooting Guide</name>
  <files>docs/TROUBLESHOOTING.md</files>
  <action>
    Create a troubleshooting reference organized by symptom category.
    Structure:
    1. Installation Issues
       - install.sh fails / permissions
       - slash commands not appearing after install
       - legacy cleanup conflicts
    2. Module Issues
       - module not loading (enabled but no hooks firing)
       - hook dispatch errors / missing refs
       - module.yaml validation
    3. State & Session Issues
       - STATE.md corrupted or out of sync
       - handoff file not found on resume
       - loop position stuck (APPLY complete but UNIFY not run)
    4. Git & GitHub Issues
       - git config not applied (pals.json git section)
       - auto-PR/push not triggering (remote != null guard)
       - branch naming conflicts
    5. Common Mistakes
       - running /paul:apply without approving plan
       - editing STATE.md manually
       - forgetting /paul:unify after apply

    Each issue: Symptom → Cause → Fix (3-line pattern).
    Keep entries concise — this is a reference, not a tutorial.
  </action>
  <verify>File exists, covers all 5 categories, each entry has symptom/cause/fix structure</verify>
  <done>AC-2 satisfied: categorized issues with symptom/cause/fix format</done>
</task>
