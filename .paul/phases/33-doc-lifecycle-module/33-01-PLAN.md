<objective>
## Goal
Create the DOCS module — Documentation Lifecycle & Oversight Companion System. A new PALS module that detects project documentation, checks for drift against code, validates README currency, and surfaces doc health during the plan-apply-unify loop.

## Purpose
PALS manages code lifecycle but has no awareness of documentation health. Projects accumulate stale READMEs, outdated API docs, and changelog gaps. DOCS brings doc awareness into the kernel loop — the same way DEAN brought dependency awareness.

## Output
- `modules/docs/module.yaml` — module manifest with 2 lifecycle hooks
- `modules/docs/references/doc-principles.md` — core philosophy and approach
- `modules/docs/references/doc-detection.md` — how to detect and catalog project docs
- `modules/docs/references/doc-drift.md` — drift detection and currency checking logic
- `pals.json` — updated with docs module entry
</objective>

    **module.yaml:** Follow DEAN's pattern exactly:
    - name: docs, display_name: DOCS
    - description: Documentation Lifecycle & Oversight Companion System. Detects project documentation, checks for drift against code changes, validates README currency, and surfaces doc health visibility.
    - version: 1.0.0, kernel: ">=0.4.0", modules: []
    - hooks:
      - pre-plan (priority 200, advisory range): Scan for project docs (README, CHANGELOG, API docs, config docs). Inject doc_warnings with stale/missing doc counts into plan context. Refs: [references/doc-detection.md]
      - post-apply (priority 250, advisory — runs after DEAN/WALT): Compare files_changed against doc file proximity. Warn if code changed but nearby docs not updated. Refs: [references/doc-drift.md]
    - files.references: list all 3 reference files
    - commands: [], platform: {}

    **doc-principles.md** (~50-60 lines):
    - Philosophy: visibility first (same as DEAN). Detect → Catalog → Check → Suggest.
    - Never auto-update docs — only surface staleness
    - Core doc types: README, CHANGELOG, API docs, architecture docs, inline code comments
    - Drift = code changed but related docs didn't
    - Currency = how recently docs were updated relative to code they describe
    - Constraints: no git blame (too slow), no content parsing (too fragile), use file modification times and proximity heuristics

    **doc-detection.md** (~100-120 lines):
    - Detection patterns for common doc files: README.md, CHANGELOG.md, CONTRIBUTING.md, docs/, API.md, ARCHITECTURE.md, *.md in project root
    - Framework-specific docs: Storybook, Swagger/OpenAPI, JSDoc config, Sphinx conf.py, MkDocs config
    - Catalog format: list of {path, type, last_modified_relative}
    - Three detection outcomes: found-standard, found-non-standard, not-found (per MODULE-SPEC section 3.8)
    - Heuristics for doc-to-code proximity: docs/ next to src/, README in package root, API docs near routes

    **doc-drift.md** (~100-120 lines):
    - Drift detection: compare files_changed (from plan) against doc catalog
    - Proximity rules: if src/auth/* changed, check docs/auth.md or README sections
    - Staleness heuristic: doc file not modified in same plan as related code = potential drift
    - Currency check: README last modified vs most recent code commit in same directory
    - Warning levels: info (docs exist, look fresh), warn (code changed, docs untouched), critical (no docs found for major component)
    - context_inject schema: doc_warnings key with {stale_count, missing_count, drift_files}
    - Avoid: git blame, content parsing, false positives on unrelated code/doc pairs
