<objective>
## Goal
Add a DEAN baseline mechanism so that pre-existing vulnerabilities are acknowledged after the first override, and subsequent plans only BLOCK if NEW critical/high vulnerabilities are detected (delta > 0 vs baseline).

## Purpose
v2.25 Item 3: DEAN blocked every plan with the same pre-existing pi-verify vulnerabilities (2 critical + 16 high). GPT-5.4 had to override 3 times across 2 loops for the exact same baseline. The override works but creates friction that trains users to reflexively skip DEAN.

## Output
Updated DEAN module with baseline-aware pre-plan hook, baseline file schema in vulnerability-audit.md, and plan-phase.md integration for baseline read/write.
</objective>

And if the baseline is older than the threshold, DEAN re-triggers full BLOCK

    Add a new `<baseline>` section to vulnerability-audit.md after the `<edge_cases>` section:

    ```markdown
    <baseline>

    ## Baseline Management

    After the first DEAN override, the current severity counts are saved to
    `.paul/dean-baseline.json`. Subsequent audits compare against this baseline.

    ### Baseline File Schema

    ```json
    {
      "created": "2026-03-26T12:00:00Z",
      "acknowledged_by": "user override during plan-phase",
      "staleness_days": 30,
      "severity": {
        "critical": 2,
        "high": 16,
        "moderate": 5,
        "low": 3
      },
      "packages": ["basic-ftp@5.0.3", "fast-xml-parser@4.2.7"]
    }
    ```

    ### Delta Comparison Logic

    On subsequent plans:
    1. Read `.paul/dean-baseline.json`
    2. Run audit command (same as normal)
    3. Compute delta: `new_critical = current.critical - baseline.critical`
    4. If `new_critical > 0` OR `new_high > 0`: BLOCK (new vulnerabilities introduced)
    5. If delta <= 0 for all severity levels: PASS with message:
       'DEAN: Baseline acknowledged ({N} critical, {M} high pre-existing). No NEW vulnerabilities.'
    6. If baseline file does not exist: use normal BLOCK logic (first run)

    ### Staleness Guard

    - Check `created` timestamp against `staleness_days` (from baseline file, default 30)
    - If baseline is older than threshold: re-trigger full BLOCK
      'DEAN: Baseline is {N} days old (threshold: {staleness_days}). Re-auditing all vulnerabilities.'
    - After new override: update baseline with fresh counts and timestamp
    - Staleness default configurable via `pals.json` `modules.dean.baseline_staleness_days` (default: 30)

    ### Clearing Baseline

    - Baseline is cleared (file deleted) on milestone completion by complete-milestone workflow
    - User can clear manually: `rm .paul/dean-baseline.json`
    - User can clear via `/paul:config` when available

    </baseline>
    ```

    Avoid: Changing existing sections — only add the new baseline section.
  </action>
