<objective>
## Goal
Create Pi-discoverable SKILL.md entry points that map PALS kernel workflows to Pi's skill system, so Pi users can invoke PALS workflows via `/skill:paul-plan`, `/skill:paul-apply`, etc.

## Purpose
Pi discovers skills by scanning for `SKILL.md` files with proper frontmatter in `~/.pi/agent/skills/`. Our installer already deploys kernel files there, but without SKILL.md files Pi can't discover or invoke them. This phase bridges that gap.

## Output
- 11 SKILL.md files in `drivers/pi/skills/{skill-name}/SKILL.md`
- 1 skill-map.md documenting workflow-to-skill mapping
- Installer will deploy these in Phase 51
</objective>

<context>
## Project Context
@.paul/PROJECT.md
@.paul/STATE.md

## Source Files
@drivers/pi/driver.yaml
@kernel/workflows/plan-phase.md (representative workflow to understand structure)

## Pi Skill Requirements
Pi skills require:
- Directory with `SKILL.md` containing frontmatter (name, description)
- Name: 1-64 chars, lowercase a-z, 0-9, hyphens only
- Description: max 1024 chars
- Progressive disclosure: name+description at startup, full content on demand
- Scripts/assets use relative paths from skill directory
- Optional: allowed-tools, disable-model-invocation
</context>

    drivers/pi/skills/paul-discuss/SKILL.md,
    drivers/pi/skills/paul-help/SKILL.md
  </files>
  <action>
    For each skill, create a SKILL.md with:

    1. **Frontmatter** (Pi required format):
       ```yaml
       ---
       name: paul-plan
       description: "Create an executable plan for the current PALS phase. Reads project state, analyzes scope, and produces a structured PLAN.md with objectives, acceptance criteria, tasks, and verification steps."
       ---
       ```

    2. **Body content** — instructions that tell Pi's LLM how to execute this PALS workflow. Each SKILL.md should:
       - State the workflow's purpose
       - List the required files to read (using `.paul/` project paths for artifacts, relative paths for kernel resources)
       - Reference the kernel workflow file for the full process: "Read the workflow definition at `../../workflows/{workflow}.md` for the complete process"
       - Reference relevant templates/references: "Read the template at `../../templates/{template}.md`"
       - Include key behavioral instructions inline (the most critical steps) so the skill works even if kernel files aren't fully loaded
       - State the expected output (what artifacts get created/modified)

    3. **Path strategy:**
       - Project artifacts: `.paul/STATE.md`, `.paul/ROADMAP.md` (absolute from project root)
       - Kernel workflows: relative from skill dir → `../../workflows/plan-phase.md`
       - Kernel references: relative from skill dir → `../../references/plan-format.md`
       - Kernel templates: relative from skill dir → `../../templates/PLAN.md`

    4. **Keep each SKILL.md concise** — Pi loads full content into context when the skill activates. Target 50-100 lines per skill. The skill points to the kernel workflow for full details rather than duplicating everything.

    5. **Do NOT add @-references** — Pi has no @ syntax. Use "Read file X" instructions instead.

    Skill-specific notes:
    - paul-help: Self-contained, lists all available /skill:paul-* commands with descriptions
    - paul-milestone: Handles both create and complete (detect from context)
    - paul-discuss: Handles both phase and milestone discussion (detect from context)
    - paul-status: Maps to /paul:progress behavior (smart status with routing)
  </action>

  </verify>
