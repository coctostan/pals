<objective>
## Goal
Create a single install.sh script that deploys PALS from a git clone to ~/.pals/ and ~/.claude/, registering commands and hooks.

## Purpose
Without an install script, PALS is just a collection of files. This makes `git clone && ./install.sh` the complete setup experience — the core "Must Have" requirement.

## Output
- `install.sh` — complete installer handling fresh install and upgrades
</objective>

<acceptance_criteria>

## AC-1: Fresh Install Works
```gherkin
Given no ~/.pals/ directory exists
When install.sh is run from the PALS repo root
Then ~/.pals/ is created with workflows/, references/, templates/, rules/, carl/
And ~/.claude/commands/pals/ contains all 25 pals commands
And ~/.claude/commands/carl/ contains all 8 carl commands
And ~/.claude/hooks/carl-hook.py exists (symlink to repo)
And ~/.claude/settings.json has CARL hook registered in UserPromptSubmit
```

## AC-2: Upgrade Preserves User Config
```gherkin
Given ~/.pals/carl/manifest already exists with user customizations
When install.sh is run again (upgrade)
Then ~/.pals/carl/manifest is NOT overwritten
And framework files (workflows, references, templates, rules) ARE updated
And hook registration is not duplicated in settings.json
```

## AC-3: Script Is Idempotent
```gherkin
Given install.sh has already been run once
When install.sh is run again
Then no errors occur
And the result is identical to a fresh install (except preserved user config)
```

</acceptance_criteria>

<task type="auto">
  <name>Task 1: Create install.sh</name>
  <files>install.sh</files>
  <action>
    Create install.sh at repo root with these sections:

    1. **Header and validation:**
       - #!/bin/bash with set -e
       - Detect PALS_ROOT (directory where install.sh lives)
       - Verify running from repo root (check for workflows/ and carl/ dirs)

    2. **Create ~/.pals/ directory structure:**
       - mkdir -p ~/.pals/{workflows,references,templates,rules,carl}
       - Copy (not symlink) framework files:
         - cp -R $PALS_ROOT/workflows/* ~/.pals/workflows/
         - cp -R $PALS_ROOT/references/* ~/.pals/references/
         - cp -R $PALS_ROOT/templates/* ~/.pals/templates/
         - cp -R $PALS_ROOT/rules/* ~/.pals/rules/

    3. **Install CARL domains:**
       - For each file in carl/: copy ONLY if destination doesn't exist (preserve user edits)
       - Use: cp -n (no-clobber) for manifest, global, context, commands, example-custom-domain
       - On upgrade: framework files update, but CARL user config preserved

    4. **Install commands:**
       - mkdir -p ~/.claude/commands/pals ~/.claude/commands/carl
       - Copy command files:
         - cp -R $PALS_ROOT/commands/pals/* ~/.claude/commands/pals/
         - cp -R $PALS_ROOT/commands/carl/* ~/.claude/commands/carl/

    5. **Install hooks:**
       - mkdir -p ~/.claude/hooks
       - Symlink: ln -sf $PALS_ROOT/hooks/carl-hook.py ~/.claude/hooks/carl-hook.py
       - chmod +x the hook script

    6. **Register hook in settings.json:**
       - If ~/.claude/settings.json doesn't exist: create with hook entry
       - If exists but no carl-hook entry: add to UserPromptSubmit array
       - If already registered: skip (idempotent)
       - Use python3 or jq for JSON manipulation (python3 more reliable)

    7. **Validation:**
       - Check all critical paths exist
       - Report success with file counts

    8. **Output summary:**
       - Print installed component counts
       - Print "PALS installed successfully" or specific errors

    Avoid: Modifying any files outside ~/.pals/ and ~/.claude/ (commands + hooks + settings only).
    Avoid: Overwriting user's CARL manifest on upgrade.
    Avoid: Complex dependency chains — keep it a simple sequential script.
  </action>
  <verify>
    - install.sh exists and is executable
    - bash -n install.sh passes (syntax check)
    - Script handles both fresh install and upgrade paths
  </verify>
  <done>AC-1, AC-2, AC-3 satisfied: Install script handles fresh install, upgrade, and idempotency</done>
</task>
