  <action>
    Add a new `## CARL Session Boundary Manager` section to drivers/pi/extensions/README.md, positioned after the "Live Module Visibility" section and before "Commands". Content should cover:

    1. **What CARL does in Pi**: Autonomous session boundary management — detects phase completion via loop signature changes and creates fresh sessions when context pressure exceeds thresholds.

    2. **Three strategies**:
       - `phase-boundary` (default): creates fresh session at phase completion only if context ratio ≥ continue_threshold
       - `always-fresh`: creates fresh session at every phase completion regardless of context
       - `manual`: disables CARL entirely; user manages sessions

    3. **Threshold defaults**: 40% continue_threshold, 80% safety_ceiling (validated in Phase 76 api-validation-report.md)

    4. **How it works**:
       - Command stashing: every /paul-* command stashes ExtensionCommandContext for later newSession() calls
       - Phase detection: agent_end compares loop signatures; "✓✓✓" means loop complete
       - Decision model: context ratio (tokens / contextWindow) compared against continue_threshold
       - Safety valve: turn_end monitors context pressure; sets pauseAtNextBoundary at safety_ceiling
       - Bootstrap: injects compact state summary + auto-runs /skill:paul-resume in new session

    5. **Configuration**: `pals.json` → `modules.carl` section with session_strategy, continue_threshold, safety_ceiling

    6. **Two-CARL model note**: Brief explanation that Claude Code CARL handles context rule injection (via UserPromptSubmit hooks), while Pi CARL handles session boundary management (via Pi extension events). They serve different roles on different platforms.

    Also update the Event Hooks section bullet for `agent_end` and `turn_end` to mention CARL wiring.

    Keep the section concise — match the existing README tone (informative, bounded, one paragraph per concept).
  </action>
