<objective>
## Goal
Design the PALS kernel/module/driver architecture — classify every existing file, define the module manifest format, specify lifecycle hook points, and document the driver interface contract.

## Purpose
This is the foundational architecture for PALS v0.4 "Kernel & Modules." Every subsequent phase depends on these decisions. The vision: PALS as the Linux of Harness Engineering — minimal kernel, modular pals, universal drivers. Getting the boundaries right now prevents rewrites later.

## Output
Four architecture documents in `docs/`:
- `ARCHITECTURE.md` — System overview (kernel, modules, drivers diagram)
- `MODULE-SPEC.md` — Module manifest format and interface contract
- `DRIVER-SPEC.md` — Driver interface contract (what a platform adapter implements)
- `KERNEL-MANIFEST.md` — Classification of every existing file as kernel/module/shared
</objective>

## AC-1: Kernel Boundary Defined
```gherkin
Given the existing 147+ framework files
When each file is classified
Then every file is labeled as kernel, module (with module name), or shared
And the kernel contains ONLY plan/apply/unify loop, state management, and lifecycle hooks
And no module-specific logic remains in kernel files
```

## AC-2: Module Manifest Format Specified
```gherkin
Given the need for self-contained, optional modules
When the module.yaml format is defined
Then it declares: name, description, version, hooks (with lifecycle points), dependencies, files, and commands
And a developer can create a new module by following the spec alone
And CARL, TODD, and WALT can each be described by this format
```

## AC-3: Lifecycle Hook Points Defined
```gherkin
Given the plan/apply/unify kernel loop
When hook points are specified
Then hooks exist at: pre-plan, post-plan, pre-apply, pre-test, post-task, post-apply, pre-unify, post-unify
And each hook defines: trigger timing, payload (what data modules receive), and return contract
And missing modules result in no-op (no errors)
```

## AC-4: Driver Interface Specified
```gherkin
Given the need to run on Claude Code, Agent SDK, and future platforms
When the driver interface is defined
Then it specifies: how to invoke a workflow, how to register hooks, how to manage state
And the Claude Code driver maps to slash commands + hook system
And the Agent SDK driver maps to system prompts + file-based orchestration
And the kernel references neither slash commands nor system prompts directly
```

<task type="auto">
  <name>Task 2: Design module manifest format and hook contract</name>
  <files>docs/MODULE-SPEC.md</files>
  <action>
    Design the module.yaml manifest format:

    ```yaml
    name: todd
    display_name: TODD
    description: Test-Driven Development enforcement
    version: 0.4.0

    hooks:
      pre-apply:
        description: Enforce RED phase — tests must be written first
        priority: 10  # lower = earlier
      post-task:
        description: Enforce GREEN phase — tests must pass
        priority: 10
      post-apply:
        description: Enforce REFACTOR phase — cleanup opportunity
        priority: 10

    dependencies:
      kernel: ">=0.4.0"
      modules: []  # no module dependencies

    files:
      workflows: [plan-phase-tdd.md]
      references: [tdd.md, tdd-detection.md, tdd-plan-generation.md, tdd-execution.md]
      commands: []  # no standalone commands

    commands: []  # slash commands this module provides
    ```

    Define all lifecycle hook points with:
    - Hook name and timing (when in the loop it fires)
    - Payload schema (what data the module receives)
    - Return contract (what the module returns — continue/block/modify)
    - Priority ordering (when multiple modules hook the same point)

    Hook points:
    - pre-plan: Before plan creation begins
    - post-plan: After plan is written, before user approval
    - pre-apply: Before apply execution starts
    - pre-test: Before test execution in a task
    - post-task: After each task completes
    - post-apply: After all tasks complete
    - pre-unify: Before unify reconciliation
    - post-unify: After loop closes

    Document how hooks compose (multiple modules on same hook point).
    Document no-op behavior when module is absent.
  </action>
  <verify>CARL, TODD, and WALT can each be fully described by the manifest format. All current integration points map to defined hooks.</verify>
  <done>AC-2 and AC-3 satisfied: Module format defined, hooks specified, composable</done>
</task>

<task type="auto">
  <name>Task 3: Design driver interface and document architecture</name>
  <files>docs/DRIVER-SPEC.md, docs/ARCHITECTURE.md</files>
  <action>
    **Driver Interface (DRIVER-SPEC.md):**

    Define what a driver must implement:

    1. workflow_invoke(workflow_name, context) — Load and execute a workflow
       - Claude Code: Maps to slash command invocation
       - Agent SDK: Loads workflow markdown as system prompt

    2. hook_register(hook_name, handler) — Register a hook handler
       - Claude Code: Wires into hooks/ system (UserPromptSubmit, etc.)
       - Agent SDK: Orchestration code calls handlers at lifecycle points

    3. state_read(file) / state_write(file, content) — State persistence
       - Claude Code: Direct filesystem (.paul/ directory)
       - Agent SDK: Could be filesystem, database, or API

    4. user_interact(prompt, options) — Get user input
       - Claude Code: Conversation-based (display text, wait for response)
       - Agent SDK: Could be API callback, webhook, or automated

    5. command_register(name, handler) — Register slash commands
       - Claude Code: Symlinks to ~/.claude/commands/
       - Agent SDK: Not applicable (no slash commands) — returns no-op

    Specify each method's signature, required vs optional, and platform mapping.

    **Architecture Overview (ARCHITECTURE.md):**

    Document the full system with ASCII diagram:
    ```
    ┌─────────────────────────────────────────┐
    │           MODULES (Pals)                │
    │  CARL  TODD  WALT  [future modules]     │
    ├─────────────────────────────────────────┤
    │           KERNEL (PAUL Core)            │
    │  Plan ──▶ Apply ──▶ Unify              │
    │  State, Lifecycle Hooks, Templates      │
    ├─────────────────────────────────────────┤
    │           DRIVERS                       │
    │  Claude Code  │  Agent SDK  │  [future] │
    └─────────────────────────────────────────┘
    ```

    Include:
    - How a workflow invocation flows through the stack
    - How a hook fires and modules respond
    - How state persists across sessions
    - How modules are loaded at install time
    - File system layout after kernelization
  </action>
  <verify>Architecture diagram is consistent with MODULE-SPEC and DRIVER-SPEC. Claude Code and Agent SDK both map cleanly to the driver interface.</verify>
  <done>AC-4 satisfied: Driver interface defined, architecture documented</done>
