<task type="auto">
  <name>Task 1: Create Reference Resolution Convention</name>
  <files>docs/REFERENCE-RESOLUTION.md</files>
  <action>
    Create docs/REFERENCE-RESOLUTION.md defining the portable reference convention:

    1. **Problem statement**: Workflows currently use Claude Code's `@path` syntax and `~/.pals/` install paths. Neither is portable.

    2. **Convention**: Define two reference categories:
       - **Project artifacts**: Plain paths relative to project root (e.g., `.paul/STATE.md`). These are already portable.
       - **Kernel resources**: Logical paths using `kernel/` prefix (e.g., `kernel/references/plan-format.md`). Adapters resolve to their install location.

    3. **In workflow files**: References appear in `<required_reading>` and `<references>` sections as plain paths without `@` prefix. The adapter's workflow loader is responsible for reading these files into context.

    4. **Harness-specific tool names**: Workflows use generic verbs:
       - "Read [path]" instead of "use the Read tool on..."
       - "Search for [pattern]" instead of "use Grep to..."
       - "Execute: [command]" instead of "use Bash to run..."
       - "Create/Edit [path]" instead of "use Write/Edit tool..."

    5. **Adapter responsibility table**: Show what each adapter does:
       | Concern | Claude Code Adapter | Pi Adapter | Generic |
       |---------|-------------------|------------|---------|
       | Load kernel resource | Reads from ~/.pals/ | Reads from ~/.pi/agent/skills/ | Reads from PALS_DIR |
       | Load project artifact | Reads from .paul/ | Reads from .paul/ | Reads from .paul/ |
       | @ syntax | Supported natively | Not applicable | Adapter-specific |

    6. **Migration examples**: Show before/after for each pattern:
       - `@~/.pals/references/plan-format.md` → `kernel/references/plan-format.md`
       - `@~/.pals/workflows/apply-phase.md` → `kernel/workflows/apply-phase.md`
       - `@~/.pals/templates/PLAN.md` → `kernel/templates/PLAN.md`
       - `@~/.pals/modules.yaml` → `kernel/modules.yaml`
       - `@.paul/STATE.md` → `.paul/STATE.md` (unchanged)

    Keep under 150 lines. This is a convention doc, not a design doc.
  </action>
  <verify>File exists, under 150 lines, covers all 6 sections above</verify>
  <done>AC-1 satisfied: Reference resolution convention documented with logical paths, adapter responsibilities, and migration examples</done>
</task>

    - "Glob tool" → "Search files:"
    - "Grep tool" → "Search content:"

    **Path changes in prose:**
    - `~/.pals/modules.yaml` → `modules.yaml` (kernel-relative)
    - `~/.pals/references/X` → logical reference only

    **DO NOT change:**
    - Workflow logic, step ordering, or behavior
    - XML tag structure (<step>, <purpose>, etc.)
    - Step names or priorities
    - Any instructions about WHAT to do (only HOW it's phrased)
    - .paul/ paths in content that gets written TO artifacts (STATE.md content etc.)

    **Verify each file preserves:**
    - Same number of steps
    - Same step names
    - Same acceptance criteria references
    - Same hook dispatch logic
