    ```
    3a. **Qualify — verify your own output (mandatory):**
        ⚠️ Your report of your own work is inherently optimistic. Trust the output, not your memory of producing it.

        a. **Re-read actual output** — open and read files you just created or modified.
           Do NOT rely on what you remember writing.
        b. **Compare against spec** — re-read the task's <action> spec line by line.
           Does the actual content satisfy every instruction?
        c. **Compare against AC** — re-read the linked acceptance criteria (Given/When/Then).
           Does the actual output satisfy the expected outcome?
        d. **If gaps found:** fix them before recording status. Do not record PASS with known gaps.

        See: references/evidence-discipline.md for anti-rationalization patterns.
    ```

<task type="auto">
  <name>Task 2: Create evidence-discipline.md reference</name>
  <files>kernel/references/evidence-discipline.md</files>
  <action>
    Create a new reference file that combines:

    1. **Evidence Chain** — the principle that no completion claim is valid without fresh verification evidence. Define the chain: Execute → Verify → Read output → Compare to spec → THEN claim.

    2. **Anti-Rationalization Table** — adapted from upstream quality-principles.md. Table of common self-deception patterns with corrective actions:
       - "Should work now" → Run verify and read output
       - "I already checked this" → Check again fresh
       - "Close enough" → Compare against AC word by word
       - "The test passes" → Also compare against the spec
       - "Minor deviation" → Log it explicitly
       - "I'm confident it works" → Run it and prove it

    3. **PALS Core Principles** (our own, not copied from upstream):
       - Plans are prompts — PLAN.md IS the prompt, not a doc transformed into one
       - Loop integrity — every PLAN closes with UNIFY, no orphan plans
       - Acceptance-driven — AC are first-class, defined before work starts
       - Evidence before claims — the qualify step exists because execution memory is optimistic
       - Scope discipline — 2-3 tasks per plan, target ~50% context
       - Modules inform, kernel decides — advisory modules surface concerns, enforcement modules gate

    Keep under 150 lines (per PALS reference file budget). Write fresh — do not copy upstream verbatim.
  </action>
  <verify>
