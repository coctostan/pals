# Plan 241-01 — Evidence Closure + Install-Surface Generalization

<objective>

## Goal

Close v2.55 by converting the Phase 239 install/uninstall one-off submodule deployment into a generalized Pi extension install surface, documenting the resulting driver.yaml decision, adding bounded validation guardrails, and preserving the next-roadmap decision in UNIFY evidence.

## Purpose

Phase 239 proved Pi extension modularization with `module-activity-parsing.ts` but left a maintainability gap: every future extracted extension submodule would require another per-file `cp` / `rm -f` edit. Phase 241 should remove that recurring install-surface drift before v2.55 closes, while preserving `.paul/*` artifact authority, parent-owned APPLY, command-output validation truth, GitHub Flow gates, no hidden state, no telemetry, no auto-approval, no auto-continue, no skipped checkpoints, no UI-only lifecycle decisions, and no inferred merge intent.

## Output

- `drivers/pi/install.sh` copies all `*.ts` files from `drivers/pi/extensions/` into `~/.pi/agent/extensions/` instead of special-casing each submodule.
- `drivers/pi/uninstall.sh` removes installed PALS extension `*.ts` files that came from the repo extension source set, without broad-deleting unrelated Pi extensions.
- Pi docs/skill map name the multi-file extension install boundary and record that `drivers/pi/driver.yaml` already delegates to the `extensions/` directory entry; no manifest edit is needed unless validation proves otherwise.
- Pi and cross-harness validation each gain at most one localized guardrail for this generalized install-surface contract.
- UNIFY summary records the v2.55 closure evidence and recommends the next roadmap focus for v2.56.

</objective>
