<objective>
## Goal
Audit every git touchpoint across PALS workflows, identify all gaps between current behavior and strict GitHub Flow, and design the pals.json config schema that will drive enforcement in Phases 85–86.

## Purpose
The requirements doc (github-flow-prompt.md) identifies that PALS treats git config as descriptive rather than enforced. Before changing workflow behavior, we need a precise inventory of what exists, what's missing, and a concrete config schema to build against.

## Output
1. `git-audit-report.md` — complete inventory of current git handling per workflow, gap analysis vs strict GitHub Flow
2. `github-flow-config-design.md` — new pals.json `git` schema with `workflow: "github-flow"` mode, field definitions, backward compatibility plan, and behavioral contract per field
</objective>

<task type="auto">
  <name>Task 2: Design the GitHub Flow config schema and behavioral contract</name>
  <files>.paul/phases/84-research-config-schema/github-flow-config-design.md</files>
  <action>
    Using the audit from Task 1, design the new pals.json git section:

    **1. New Config Schema:**
    Define the full `git` section with all fields needed for GitHub Flow mode:
    ```json
    "git": {
      "workflow": "github-flow | legacy | none",
      "remote": "...",
      "base_branch": "main",
      "merge_method": "squash | merge | rebase",
      "auto_push": true/false,
      "auto_pr": true/false,
      "ci_checks": true/false,
      "delete_branch_on_merge": true/false,
      "update_branch_when_behind": true/false,
      "require_pr_before_next_phase": true/false,
      "require_reviews": false
    }
    ```

    For each field: type, default, what happens when true vs false, which workflow steps read it.

    **2. Behavioral Contract per workflow:**
    For each workflow that will change in Phases 85–86, document:
    - What new behavior is triggered by `workflow: "github-flow"`
    - What existing behavior is preserved for `workflow: "legacy"` (maps to current `branching: "feature-per-phase"`)
    - What the `workflow: "none"` mode does (no git ops, replaces missing config)

    **3. Backward Compatibility:**
    - How existing `branching: "feature-per-phase"` maps to `workflow: "legacy"`
    - How existing `branching: "direct-to-main"` is handled
    - Migration path for existing pals.json files

    **4. Key Design Decisions:**
    - Where the merge gate lives (unify-phase vs transition-phase)
    - What "require PR before next phase" actually blocks
    - How CI state is checked (gh CLI, polling, or advisory)
    - Branch naming convention (keep `feature/{phase}` or change)

    Avoid: Writing workflow implementations — this is schema + contract only.
  </action>
  <verify>
    - Schema covers all fields from github-flow-prompt.md requirements
    - Every field has type, default, and behavioral description
    - Backward compatibility section addresses existing config migration
    - Contract maps each workflow to its new/changed behavior under github-flow mode
    - Design decisions are explicit (not left ambiguous for implementation)
  </verify>
  <done>AC-3 satisfied: Config schema designed with behavioral contract, backward compatibility, and explicit design decisions</done>
