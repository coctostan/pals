<objective>
## Goal
Surface the v2.51 validation-classification model in user-facing docs, Pi-facing docs, and historical validation-report artifacts, then add bounded validation guardrails that prove those surfaces remain non-authoritative and subordinate to fresh command output.

## Purpose
Phase 218 established the classification contract and Phase 219 made the validation-suite inventories command-output-backed. Phase 220 should make that model discoverable without letting stale reports, summaries, widgets, or Pi surfaces become a validation ledger. Fresh validation command output remains the only pass/fail truth; documentation and report artifacts explain posture and routing only.

## Output
- README and Pi documentation explain validation classes, current command-output baseline, and non-authoritative report surfacing.
- Historical validation reports clearly identify their old counts as archival and point to current command-output truth.
- The validation classification contract records Phase 219 hardening evidence and Phase 220 surfacing expectations.
- Pi validation increases from 201→202 and cross-harness validation increases from 117→118 only because each suite gains one localized documentation/report surfacing guardrail.
</objective>
