<objective>
## Goal
Define a repo-controlled PALS-native implementer design for the `@tintinweb/pi-subagents` extension that uses Pi subagents deliberately while preserving inline APPLY as the behavioral reference and `.paul/*` as the authoritative lifecycle layer.
## Purpose
Phase 154 established that current generic `implementer` / `worker` behavior is not a proven PALS APPLY path. This phase must turn that audit into an explicit design contract for `@tintinweb/pi-subagents` so Phase 156 can implement deliberate routing using the extension's `.md` agent format and `Agent()` tool calling without guessing about prompt assembly, module reach, verification ownership, or fallback behavior.

## Output
Three design artifacts in `.paul/phases/155-pals-implementer-design/`:
- `pals-implementer-contract.md` — parent/subagent responsibility model, lifecycle authority, fallback, and equivalence rules
- `prompt-and-module-strategy.md` — prompt assembly, bounded context, module participation, tooling, and verification boundaries
- `integration-handoff.md` — concrete implementation targets for Phase 156 and validation targets for Phase 157
</objective>
