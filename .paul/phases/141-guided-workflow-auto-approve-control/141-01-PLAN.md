<objective>
## Goal
Add per-transition-type auto-approve configuration to the Pi guided workflow so users retain control at critical decision points (plan approval, phase transitions) while allowing routine transitions (continue to UNIFY) to auto-present.

## Purpose
E2E testing (v2.29 Item 1 — High) found that the guided workflow auto-cascades through multiple approval checkpoints without user control, bypassing review at plan approval, UNIFY transitions, and phase boundaries. Users need the ability to pause at critical decision points while still benefiting from guided workflow assistance at routine transitions.

## Output
- `pals.json` gets a `guided_workflow` config section with per-type auto-approve settings
- `pals-hooks.ts` loads config and respects it when presenting guided workflow moments
- `init-project.md` sets safe defaults for new projects
- Reinstall to deployed location
</objective>

<task type="auto">
  <name>Task 1: Add guided_workflow config loading and per-type gating to pals-hooks.ts</name>
  <files>drivers/pi/extensions/pals-hooks.ts, pals.json</files>
  <action>
    1. Add a GuidedWorkflowConfig type with per-transition-type `auto_present` boolean map:
       ```typescript
       type GuidedWorkflowConfig = {
         auto_present: {
           plan_review: boolean;        // default: false (critical — user should review)
           apply_approval: boolean;     // default: false (critical — user decides to execute)
           continue_to_unify: boolean;  // default: true  (routine)
           checkpoint_decision: boolean; // default: true  (interactive by nature)
           checkpoint_human_verify: boolean; // default: true
           checkpoint_human_action: boolean; // default: true
           resume_next: boolean;        // default: true  (routine)
           phase_transition: boolean;   // default: false (critical — user decides next phase)
           milestone_transition: boolean; // default: false (critical — user decides direction)
         };
       };
       ```
    2. Add a `loadGuidedWorkflowConfig(cwd: string)` function that reads pals.json `guided_workflow.auto_present` with the defaults above. Missing keys fall back to defaults. Missing entire section falls back to all defaults.
    3. Add a constant `GUIDED_WORKFLOW_DEFAULTS` for the default auto_present map.
    4. Modify `presentGuidedWorkflowMoment` to accept config and check `auto_present[moment.kind]`:
       - If `auto_present` is `true` for this kind → present the interactive dialog (current behavior)
       - If `auto_present` is `false` → show an info notification with the summary + instruction to respond in the canonical prompt, and do NOT call `ctx.ui.confirm()` or `ctx.ui.select()` or `sendCanonicalWorkflowResponse()`
    5. Update `maybePresentGuidedWorkflow` (inside `palsHooks()`) to load config and pass it.
    6. Add `guided_workflow` section to pals.json with the default config:
       ```json
       "guided_workflow": {
         "auto_present": {
           "plan_review": false,
           "apply_approval": false,
           "continue_to_unify": true,
           "checkpoint_decision": true,
           "checkpoint_human_verify": true,
           "checkpoint_human_action": true,
           "resume_next": true,
           "phase_transition": false,
           "milestone_transition": false
         }
       }
       ```

    IMPORTANT: The kind values in GuidedWorkflowMoment use hyphens ("plan-review") but pals.json keys use underscores ("plan_review"). The config loader must map between them (replace hyphens with underscores when looking up config).

    Avoid: Changing any existing guided workflow detection logic. Only gate the presentation behavior.
    Avoid: Adding Pi-owned workflow state. Config is in shared pals.json.
  </action>
