## AC-2: Source-to-Destination File Map
```gherkin
Given the 111 source files identified in the architecture audit
When the structure design maps each file
Then every source file has exactly one destination path in the PALS structure
And files with no destination are explicitly marked "remove" or "merge into X" with rationale
```

## AC-3: Namespace and Naming Conventions
```gherkin
Given that PALS unifies four component namespaces (paul, carl, todd, walt)
When naming conventions are defined
Then the design specifies: file naming pattern, command namespace (/pals:*), and how component identity is preserved or merged
And the conventions are consistent across all file categories (workflows, references, templates, rules, commands)
```

  <files>.paul/phases/01-architecture-design/pals-structure-design.md</files>
  <action>
    Create the structure design document with these sections:

    1. **Target Root** — Define the installation root (e.g., ~/.pals/) and rationale

    2. **Directory Tree** — Full target directory layout:
       - Core workflows (PAUL lifecycle: plan, apply, unify, etc.)
       - TDD workflows (TODD, merged into core or subfolder)
       - Quality workflows/references (WALT, merged into core or subfolder)
       - Context/rules system (CARL domains, manifest, hooks)
       - Templates (all template files)
       - References (all reference docs)
       - Rules (framework rules)
       - Commands/skills (slash commands)

    3. **Naming Conventions:**
       - File naming pattern for each category
       - Command namespace: /pals:* replacing /paul:* and /carl:*
       - How TODD/WALT identity is handled (named sections vs separate dirs)

    4. **Source-to-Destination Map** — Table for all 111 files:
       - Source path → Destination path
       - Action: copy | rename | merge | remove
       - Notes for any non-obvious decisions

    5. **Key Design Decisions** — Document choices with rationale:
       - Flat vs nested for TODD/WALT files
       - CARL placement (subdirectory vs peer)
       - Command consolidation approach
       - What gets removed vs preserved

    6. **Migration Notes** — Anything Phase 2+ needs to know:
       - Reference rewiring needed (@-paths that change)
       - Files that need content updates (not just moves)
       - Order-dependent migrations

    Use the architecture audit as the source of truth for the file inventory.
    Ensure the design accounts for the key decisions from 01-01:
    - CARL is independent (hook-level integration)
    - TODD/WALT are embedded subsets (not overlays)
