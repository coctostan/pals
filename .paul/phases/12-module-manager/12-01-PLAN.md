<objective>
## Goal
Refactor install.sh to be module-aware: read module.yaml manifests dynamically instead of hardcoding TODD/WALT/CARL, generate a modules.yaml registry of installed modules, and create a pals.yaml config file for users to control which modules are active.

## Purpose
Completes the kernel/module decoupling started in Phases 10-11. The kernel workflows already dispatch hooks via modules.yaml — but that file doesn't exist yet because the installer still hardcodes module installation. This plan closes that gap.

## Output
- `install.sh` — refactored to read module.yaml manifests and install modules dynamically
- `pals.yaml` — default configuration file listing active modules
- `~/.pals/modules.yaml` — generated at install time, tracks installed modules + hooks
</objective>

## AC-1: Dynamic Module Discovery
```gherkin
Given the installer runs from the PALS repo root
When modules/ contains subdirectories with module.yaml files
Then install.sh discovers and installs each module by reading its manifest
  And no module names are hardcoded in install.sh (except in legacy cleanup)
```

## AC-2: Module Files Installed Per Manifest
```gherkin
Given a module.yaml declares files.references, files.workflows, files.config, commands, and platform.claude_code.hooks
When that module is installed
Then each declared file category is copied to the correct installed location:
  - references/* → ~/.pals/references/
  - workflows/* → ~/.pals/workflows/
  - config/* → ~/.pals/<module-name>/ (preserve existing, don't overwrite)
  - commands → ~/.claude/commands/<command-namespace>/
  - platform hooks → symlinked + registered in settings.json
```

## AC-3: modules.yaml Generated
```gherkin
Given install.sh completes successfully
When all modules are installed
Then ~/.pals/modules.yaml exists with:
  - kernel_version field
  - installed_modules map with each module's name, version, installed hooks, and timestamp
  And the format matches MODULE-SPEC.md section 5.3
```

<task type="auto">
  <name>Task 2: Refactor install.sh for dynamic module installation</name>
  <files>install.sh</files>
  <action>
    Rewrite install.sh to replace all hardcoded module installation with manifest-driven logic. The overall structure becomes:

    **Section 0 (keep): Legacy cleanup** — unchanged

    **Section 1 (keep): Install kernel files** — unchanged (kernel/ directory structure is stable)

    **Section 2 (rewrite): Install modules dynamically**
    Replace the hardcoded TODD/WALT/CARL blocks with a loop that:
    1. Reads pals.yaml to get list of active modules (default: all active if no pals.yaml)
    2. For each active module, reads its module.yaml manifest
    3. Installs files per manifest declarations:
       - `files.references` → cp to ~/.pals/references/
       - `files.workflows` → cp to ~/.pals/workflows/
       - `files.templates` → cp to ~/.pals/templates/
       - `files.rules` → cp to ~/.pals/rules/
       - `files.config` → cp to ~/.pals/<module-name>/ (skip if dest exists — preserve user config)
    4. Tracks installed modules for modules.yaml generation

    **Section 3 (rewrite): Install commands dynamically**
    Replace hardcoded paul/carl command copying with:
    1. Kernel commands: cp kernel/commands/paul/ → ~/.claude/commands/paul/ (unchanged)
    2. Module commands: for each active module with `commands` in manifest, determine command namespace from command name prefix and copy command files to ~/.claude/commands/<namespace>/

    **Section 4 (rewrite): Install hooks dynamically**
    Replace hardcoded carl-hook.py with:
    1. For each active module with `platform.claude_code.hooks`, symlink source file and register in settings.json
    2. Use the existing Python script approach for settings.json registration, but parameterize the hook command and source path

    **Section 5 (new): Generate modules.yaml**
    After all modules installed, write ~/.pals/modules.yaml:
    ```yaml
    kernel_version: "0.4.0"
    installed_modules:
      <module-name>:
        version: "<from module.yaml>"
        installed_at: "<ISO timestamp>"
        hooks: [<lifecycle hook names from module.yaml>]
        platform_hooks: [<platform hook events, if any>]
    ```
    Use a Python heredoc (like the existing settings.json approach) to parse module.yaml files and generate this.

    **Section 6 (update): Validation**
    Update validation to check dynamically — iterate installed modules instead of hardcoding paths.

    **Section 7 (update): Summary**
    Show per-module install status.

    **Important implementation notes:**
    - Use python3 for YAML parsing (module.yaml files) — it's already a dependency (used for settings.json)
    - PyYAML may not be installed. Use a simple line-based parser or inline Python that doesn't need PyYAML. Parse the YAML manually with basic string processing since module.yaml files have a known, flat-ish structure.
    - Actually: the simplest approach is to use python3 with a minimal YAML subset parser. The module.yaml files are simple enough that we can parse key-value pairs and list items without a full YAML library. Write a small Python function inline.
    - Maintain backward compatibility: if pals.yaml doesn't exist, install all modules (same as current behavior)
    - CARL config preservation behavior must be maintained (skip existing config files)
  </action>
  <verify>

    3. Verify ~/.pals/modules.yaml exists and contains all 3 modules with correct hooks
    4. Verify all files that were installed before are still installed (no regression)
    5. Verify ~/.pals/carl/ config files preserved if they existed
  </verify>
