```gherkin

Then each module's installed files are removed per its manifest declarations
  And kernel files (~/.pals/workflows, references, templates, rules) are removed
  And kernel commands (~/.claude/commands/paul/) are removed
  And platform hooks are deregistered from settings.json
  And modules.yaml itself is removed
  And no hardcoded module names appear in the removal logic
```

## AC-3: Single Module Install
```gherkin
Given a module exists in modules/ with a module.yaml manifest
  And that module is not currently installed (or needs updating)
When the user runs `./pals install <module-name>`
Then the module's files are installed per manifest
  And modules.yaml is regenerated to include the module
  And the user sees confirmation of what was installed
```

```gherkin
Given a module is currently installed (listed in modules.yaml)
When the user runs `./pals remove <module-name>`
Then the module's installed files are removed (references, workflows, commands, hooks)
  And modules.yaml is regenerated without the module
  And platform hooks for that module are deregistered from settings.json
  And the user sees confirmation of what was removed
```

<task type="auto">
  <name>Task 1: Refactor uninstall.sh for module-aware cleanup</name>
  <files>uninstall.sh</files>
  <action>
    Rewrite uninstall.sh to read modules.yaml and remove per-module files dynamically.

    **Structure:**

    1. Check if installed (keep existing check)

    2. Read modules.yaml — for each installed module:
       - Read its module.yaml from the PALS repo (if available) or use modules.yaml metadata
       - Remove module files from installed locations:
         - References: find and remove module-contributed files from ~/.pals/references/
         - Workflows: find and remove module-contributed files from ~/.pals/workflows/
         - Config: remove ~/.pals/<module-name>/ directory
         - Commands: remove ~/.claude/commands/<module-namespace>/
         - Platform hooks: remove symlinks from ~/.claude/hooks/, deregister from settings.json

    3. Remove kernel files:
       - rm -rf ~/.pals/ (this catches everything including modules.yaml)

    4. Remove kernel commands:
       - rm -rf ~/.claude/commands/paul/

    5. Legacy cleanup (keep existing)

    6. Summary

    **Implementation approach:**
    Use Python (like install.sh) to read modules.yaml and settings.json.
    The key difference from install.sh: uninstall reads the installed modules.yaml
    (at ~/.pals/modules.yaml) rather than the source manifests. But for file-level
    cleanup, we need to know which files belong to which module — this comes from
    reading the source module.yaml manifests.

    **Simpler approach:** Since uninstall removes ALL of ~/.pals/ anyway, we only need
    module-awareness for:
    - Removing module commands from ~/.claude/commands/<namespace>/
    - Removing module hooks from ~/.claude/hooks/
    - Deregistering hooks from settings.json

    For the full ~/.pals/ directory, rm -rf is still correct (removes everything).
    The per-module awareness matters for ~/.claude/ artifacts.

    Use a Python block that:
    1. Reads ~/.pals/modules.yaml (if exists)
    2. For each module, checks for commands → removes ~/.claude/commands/<namespace>/
    3. For each module, checks for platform hooks → removes symlinks, deregisters from settings.json
    4. Falls through to shell for rm -rf ~/.pals/ and rm -rf ~/.claude/commands/paul/

    **Important:** Keep legacy cleanup section. Keep stale /pals namespace cleanup.
    No hardcoded module names in the dynamic removal logic.
  </action>
  <verify>
    1. Read uninstall.sh and confirm no hardcoded module names in removal logic (legacy cleanup excluded)
    2. Verify the script handles missing modules.yaml gracefully (first-time uninstall)
  </verify>
  <done>AC-1 satisfied: uninstall reads modules.yaml dynamically for per-module cleanup</done>
</task>

<task type="auto">
  <name>Task 2: Create pals CLI with modules/install/remove subcommands</name>
  <files>pals</files>
  <action>
    Create a `pals` bash script in the repo root. This is a lightweight CLI for module management.

    **Subcommands:**

    `./pals modules` (or `./pals list`):
    - Read ~/.pals/modules.yaml
    - Display table: Name | Version | Lifecycle Hooks | Platform Hooks | Status
    - Status: "installed" for all listed modules
    - Also scan modules/ directory to show available-but-not-installed modules
    - If modules.yaml doesn't exist: "PALS not installed. Run ./install.sh first."

    `./pals install <module-name>`:
    - Validate module exists in modules/ directory with module.yaml
    - Read the module.yaml manifest
    - Install files per manifest (same logic as install.sh Python block — reuse pattern)
    - Regenerate modules.yaml with the new module added
    - If module already installed: "Module <name> already installed. Run ./install.sh to update."

    `./pals remove <module-name>`:
    - Validate module is installed (exists in modules.yaml)
    - Read module.yaml manifest to determine what files to remove
    - Remove module files:
      - References contributed by this module from ~/.pals/references/
      - Workflows contributed from ~/.pals/workflows/
      - Config directory ~/.pals/<module-name>/
      - Commands from ~/.claude/commands/<namespace>/
      - Platform hooks: remove symlink, deregister from settings.json
    - Regenerate modules.yaml without the module
    - Cannot remove kernel (only modules)

    `./pals` (no args) or `./pals help`:
    - Show usage: available subcommands with brief descriptions

    **Implementation:**
    - Bash wrapper that dispatches to Python blocks for YAML parsing/file operations
    - Reuse the parse_module_yaml function from install.sh (copy it — don't import)
    - Make executable: chmod +x pals

    **Important:** The pals script operates from the repo root (needs access to modules/ directory for manifests). It is NOT installed to PATH — users run it as ./pals from the repo.
  </action>
  <verify>
    1. chmod +x pals && ./pals help — shows usage
    2. ./pals modules — shows installed modules table
    3. Verify ./pals install and ./pals remove have proper validation (error messages for missing args, unknown modules)
  </verify>
  <done>AC-2, AC-3, AC-4 satisfied: pals CLI provides module listing, install, and remove</done>
</task>
