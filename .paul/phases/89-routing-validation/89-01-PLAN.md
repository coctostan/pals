<objective>
## Goal
Add explicit behind-base routing priority to resume's git-aware routing table. Upgrade init's GitHub CLI validation from presence-only (`gh --version`) to authentication-aware (`gh auth status`).

## Purpose
GitHub Flow correctness requires that stale branches are surfaced as actionable blockers — not passive data in a display box. And init must not declare GitHub Flow "ready" when `gh` is installed but unauthenticated, since PR/check/merge commands will fail immediately.

## Output
2 updated kernel workflow files with behind-base routing priority and auth-aware CLI validation.
</objective>

## AC-1: Behind-base routing is explicit and prioritized
```gherkin
Given a GitHub Flow project where the feature branch is behind the base branch
When the user resumes work
Then the git-aware routing table suggests "Update branch from base" as the next action, before suggesting implementation work or PR merge
```

## AC-2: Behind-base takes precedence over merge readiness
```gherkin
Given a GitHub Flow project where the branch is behind base AND has an open PR with CI passing
When resume routing evaluates git state
Then behind-base routing fires first (higher priority than "ready to merge"), because merging a stale branch can cause CI failures or conflicts
```

## AC-3: Init validates gh authentication, not just CLI presence
```gherkin
Given a user selecting GitHub Flow during init
When the init workflow validates gh CLI availability
Then it runs `gh auth status` (not just `gh --version`) and distinguishes "CLI missing" from "CLI present but not authenticated"
```

## AC-4: Init messaging is clear about auth failure
```gherkin
Given gh CLI is installed but `gh auth status` fails
When the user selects GitHub Flow
Then the warning message says authentication is missing (not just "gh not found") and explains that PR/CI/merge operations will fail
```

  <action>
    In the `determine_single_action` step (around line 136), update the git-aware routing table to add behind-base as a priority entry.

    **Current routing table (4 entries):**
    ```
    | Git State | Override Next Action |
    |-----------|---------------------|
    | PR open + CI failing | "Fix CI failures, then merge PR" |
    | PR open + CI passing + reviews pending | "Get PR reviewed" |
    | PR open + CI passing + ready to merge | "Merge PR: `gh pr merge`" |
    | No PR + loop complete (all ✓) | Normal routing |
    ```

    **Replace with (5 entries, behind-base inserted at position 2):**
    ```
    | Git State | Override Next Action |
    |-----------|---------------------|
    | PR open + CI failing | "Fix CI failures, then merge PR" |
    | BEHIND_COUNT > 0 (branch behind base) | "Update branch from base: `git fetch origin {GIT_BASE_BRANCH} && git rebase origin/{GIT_BASE_BRANCH}` then re-push and recheck CI" |
    | PR open + CI passing + reviews pending (if `require_reviews: true`) | "Get PR reviewed" |
    | PR open + CI passing + ready to merge | "Merge PR: `gh pr merge`" |
    | No PR + loop complete (all ✓) | Normal routing (next PLAN will create branch) |
    ```

    **Rationale for position 2 (after CI failing, before reviews/merge):**
    - CI failure is always the highest priority (can't do anything useful until CI passes)
    - Behind-base is next because merging a stale branch can cause post-merge CI failures or conflicts, so updating before merge/review is the correct sequence
    - Review/merge routing only makes sense once the branch is current with base

    **Also add a brief note after the table:**
    ```
    Note: BEHIND_COUNT > 0 fires even if the PR is ready to merge. Merging a behind-base
    branch risks post-merge CI failures. Always update from base first.
    ```

    **Do NOT change:**
    - The `check_git_state` step (data collection is already correct — BEHIND_COUNT is already gathered)
    - The `report_and_route` step template (the display already shows behind/ahead count)
    - Any non-GitHub-Flow routing logic
  </action>

  <files>kernel/workflows/init-project.md</files>
  <action>
    In the `configure_git` step (around line 606-612), replace the gh CLI detection with a two-tier check.

    **Current (line 608-612):**
    ```
    **0. Detect `gh` CLI availability:**
    ```bash
    gh --version 2>/dev/null
    ```
    - Store `gh_available = true/false` for later use
    ```

    **Replace with:**
    ```
    **0. Detect `gh` CLI availability and authentication:**
    ```bash
    gh --version 2>/dev/null
    ```
    - If command fails: `gh_available = false`, `gh_authenticated = false`
    - If command succeeds: `gh_available = true`, then check authentication:
    ```bash
    gh auth status 2>/dev/null
    ```
    - If auth check succeeds: `gh_authenticated = true`
    - If auth check fails: `gh_authenticated = false`
    - Store both `gh_available` and `gh_authenticated` for later use
    ```

    **Then update the GitHub Flow selection warning (around line 653-661).** Currently it only checks `gh_available`:

    **Current:**
    ```
    **If "1" (GitHub Flow):**
    - If `gh_available = false`:
      ```
      ⚠️  GitHub Flow requires the `gh` CLI for PR/CI operations.
      Install: https://cli.github.com/

      [1] Continue anyway (enforcement features will be limited)
      [2] Choose a different workflow
      ```
    ```

    **Replace with a two-tier warning:**
    ```
    **If "1" (GitHub Flow):**
    - If `gh_available = false`:
      ```
      ⚠️  GitHub Flow requires the `gh` CLI for PR/CI operations.
      Install: https://cli.github.com/

      [1] Continue anyway (enforcement features will be limited)
      [2] Choose a different workflow
      ```
    - If `gh_available = true` but `gh_authenticated = false`:
      ```
      ⚠️  `gh` CLI is installed but not authenticated.
      GitHub Flow requires authentication for PR creation, CI checks, and merging.
      Run: `gh auth login`

      [1] Continue anyway (enforcement features will fail until authenticated)
      [2] Choose a different workflow
      ```
    ```
