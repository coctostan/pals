<objective>
## Goal
Create the IRIS (Intelligent Review & Inspection System) module as the fifth first-party pal. IRIS provides structured code review guidance and inspection checklists, surfaced through a `/paul:review` command.

## Purpose
Code review is the last major gap in PALS' development lifecycle coverage. CARL manages context rules, TODD enforces TDD, WALT gates quality, DEAN audits dependencies — but nothing guides the review and inspection of code changes. IRIS fills this gap with structured review patterns, common code smell detection guidance, and inspection checklists tailored by change type. This also completes the v0.6 milestone, proving the module system scales to 5 pals.

## Output
- `modules/iris/module.yaml` — module manifest (no hooks in v0.6 — read-only visibility first)
- `modules/iris/references/review-principles.md` — philosophy and approach to code review
- `modules/iris/references/review-patterns.md` — common patterns to look for during review (code smells, anti-patterns, security concerns)
- `modules/iris/references/inspection-checklists.md` — structured checklists by change type (feature, bugfix, refactor, config)
- `kernel/commands/paul/review.md` — the `/paul:review` command (generates review guidance for recent changes)
</objective>

    - Philosophy: guide reviewers with structure, not replace judgment
    - Core principle: detect change type → select checklist → surface relevant patterns
    - Review scope: focused reviews (what changed) not exhaustive audits
    - Tone guidance: constructive, specific, actionable feedback

    Structure with clear categories:
    - **Code Smells:** duplication, long methods/functions (>40 lines), deep nesting (>3 levels), feature envy, god objects/files, primitive obsession, shotgun surgery
    - **Security Concerns:** SQL/command injection, XSS, hardcoded secrets/credentials, unsafe deserialization, missing input validation, insecure randomness, path traversal
    - **Performance Anti-patterns:** N+1 queries, unnecessary allocations in loops, blocking I/O in async contexts, missing pagination, unbounded queries, excessive logging
    - **Maintainability Issues:** missing error handling, tight coupling, magic numbers/strings, unclear naming, commented-out code, TODO without tracking, missing types/interfaces
    Each pattern: name, what to look for (detection heuristic), why it matters, suggested fix (brief)

    **4. Create inspection-checklists.md** (checklists by change type):
    Structure with checklist sections:
    - **Feature checklist:** correctness, edge cases, error handling, security, tests, docs, backwards compatibility
    - **Bugfix checklist:** root cause identified, regression test added, related code checked, fix is minimal
    - **Refactor checklist:** behavior preserved, tests still pass, no scope creep, readability improved
    - **Config/infra checklist:** secrets not committed, environment parity, rollback plan, monitoring
    - **General (all types):** naming, readability, test coverage, no debug artifacts, commit message quality
    Each item: checkbox format with brief description

<task type="auto">
  <name>Task 2: Create /paul:review command</name>
  <files>kernel/commands/paul/review.md</files>
  <action>
    Create the command file following deps.md and coverage.md patterns (frontmatter + process steps).

    Frontmatter:
    - name: pals:review
    - description: Generate structured code review guidance for recent changes
    - argument-hint: "[optional: commit range, file path, or 'staged']"
    - allowed-tools: [Read, Bash, Glob, Grep]

    Command workflow:

    1. **Determine review scope**:
       - If $ARGUMENTS = "staged": review staged changes (`git diff --cached`)
       - If $ARGUMENTS = commit hash or range: review that range (`git diff {range}`)
       - If $ARGUMENTS = file path: review changes in that file
       - If no $ARGUMENTS: review all uncommitted changes (`git diff` + `git diff --cached`)
       - If no changes found: display helpful message (AC-5)

    2. **Analyze changes**:
       - Run `git diff --stat` for the determined scope to get file list
       - Run `git diff` for actual content changes
       - Categorize changed files by type (source, test, config, docs)
       - Detect change type heuristic:
         - New files → feature
         - Test files only → test improvement
         - Config files → config/infra
         - Small focused change → likely bugfix
         - Large restructuring → refactor
         - Mixed → feature (default)

    3. **Select applicable patterns** (per review-patterns.md):
       - Based on file types changed (e.g., .sql files → check SQL injection)
       - Based on change size (large changes → check for god objects)
       - Based on language patterns detected in diff

    4. **Generate checklist** (per inspection-checklists.md):
       - Select primary checklist based on detected change type
       - Include general checklist items always
       - Highlight items most relevant to the specific changes

    5. **Present review dashboard**:
       ```
       ════════════════════════════════════════
       IRIS CODE REVIEW
       ════════════════════════════════════════

       Scope: {description of what's being reviewed}
       Files: {N} changed ({added} added, {modified} modified, {deleted} deleted)
       Change Type: {feature|bugfix|refactor|config} (detected)

       ── Files Changed ──────────────────────
       {categorized file list with +/- line counts}

       ── Review Patterns to Check ───────────
       {applicable patterns based on changes, each with brief description}

       ── Inspection Checklist ────────────────
       {relevant checklist items as checkboxes}

       ── Specific Suggestions ────────────────
       {targeted suggestions based on actual diff content}
       {e.g., "New API endpoint in routes.ts — verify input validation"}
       {e.g., "Database query added — check for N+1 pattern"}

       ════════════════════════════════════════
       ```

    6. **No changes handling** (AC-5):
       ```
       ════════════════════════════════════════
       IRIS CODE REVIEW
       ════════════════════════════════════════

       No changes detected in working tree.

       Usage:
         /paul:review              — Review all uncommitted changes
         /paul:review staged       — Review staged changes only
         /paul:review HEAD~3..HEAD — Review last 3 commits
         /paul:review src/auth/    — Review changes in specific path

       IRIS supports: code smells, security concerns,
       performance anti-patterns, and change-type checklists.
       ════════════════════════════════════════
       ```

    Error handling:
    - Not a git repo → "IRIS requires git. Initialize with `git init`."
    - Invalid commit range → "Invalid range: {input}. Use format: commit1..commit2"
    - Binary files → skip with note "Binary files excluded from review"
  </action>
