<objective>
## Goal
Add bounded Pi-native guided workflow UX for canonical PALS lifecycle moments so approvals, checkpoints, resume routing, and phase/milestone transitions feel more natural inside Pi without moving workflow ownership out of shared artifacts and shared markdown prompts.

## Purpose

<boundaries>
## DO NOT CHANGE
- `.paul/*` and shared kernel workflow markdown as the authoritative lifecycle model
- Canonical `/skill:paul-*` workflow ownership or cross-harness shared workflow semantics
- Non-Pi drivers or shared kernel behavior outside normal PLAN/APPLY/UNIFY artifact updates
- The bounded activation/injection model established in Phase 62, except where guidance needs to read existing state or canonical prompts
- Milestone-wide readability/color polish, help-surface alignment, and installation-strategy clarification reserved for Phase 64

## SCOPE LIMITS
- This plan adds guided Pi UX for approvals, checkpoints, resume routing, and phase/milestone transitions only
- Keep the detection surface bounded to standard PALS workflow markers and current lifecycle context; do not build a generic conversation-understanding layer
- Do not add Pi-only persistent workflow state (`appendEntry`, hidden state machines, or adapter-owned lifecycle checkpoints)
- Do not auto-execute APPLY, UNIFY, next-phase planning, or milestone transitions without explicit user interaction in the current session
- Do not expand the shortcut layer beyond what is genuinely needed to support the guided workflow UX in this phase
</boundaries>
