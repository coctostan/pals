<objective>
## Goal
Create and run a cross-harness validation suite that proves the same `.paul/` project can be driven by both Claude Code and Pi adapters with artifact parity, lifecycle fidelity, and module participation.

## Purpose
This is the final gate for v2.0 Portability & Cross-Harness. Without validation, the portable reference convention (Phase 46), specs (Phase 45), and Pi adapter (Phase 47) are unproven claims. This phase proves they work.

## Output
- `tests/cross-harness-validation.sh` — executable validation script
- `docs/VALIDATION-REPORT.md` — results documenting pass/fail for each check
</objective>

## AC-1: Installer Parity
```gherkin
Given the PALS repo with both claude-code and pi drivers
When the root installer runs with each driver individually (PALS_DRIVER=claude-code, PALS_DRIVER=pi)
Then both installations complete without errors
And both produce a valid modules.yaml with the same module set
And both contain the same kernel file count (workflows, references, templates, rules)
```

## AC-2: Portability Compliance
```gherkin
Given all kernel workflow, reference, template, and rule files
When scanned for harness-specific references
Then zero files contain @~/.pals/, @.paul/, ~/.pals/ hardcoded paths, or Claude Code tool-specific language (e.g., "Use the Read tool", "Use Bash to")
And all references use the portable kernel/ or .paul/ prefix convention per REFERENCE-RESOLUTION.md
```

## AC-3: Artifact Spec Compliance
```gherkin
Given the current .paul/ project artifacts (STATE.md, PROJECT.md, ROADMAP.md)
When validated against docs/ARTIFACT-SPEC.md required sections
Then each artifact contains all required sections
And STATE.md has: Project Reference, Current Position, Loop Position, Accumulated Context, Session Continuity
And PROJECT.md has: Description, Core Value, Requirements
And ROADMAP.md has: Overview, Milestones, Current Milestone with phase table
```

## AC-4: Driver Manifest Completeness
```gherkin
Given drivers/claude-code/driver.yaml and drivers/pi/driver.yaml
When validated against docs/ADAPTER-CONTRACT.md required capabilities
Then each driver declares all required capability fields (workflow_invoke, state_read, state_write, user_interact)
And each driver has install.sh and uninstall.sh
And capability differences are documented (Pi: no hook_register, no command_register)
```

## AC-5: Validation Report
```gherkin
Given all validation checks have run
When results are compiled
Then docs/VALIDATION-REPORT.md contains a pass/fail table for each check category
And the report includes the validation date and commit hash
And all checks pass (zero failures)
```

<task type="auto">
  <name>Task 1: Create cross-harness validation script</name>
  <files>tests/cross-harness-validation.sh</files>
  <action>
    Create a bash validation script with these check categories:

    **1. Installer Parity Checks:**
    - Run `PALS_DRIVER=claude-code bash install.sh` in a temp environment, count installed kernel files
    - Run `PALS_DRIVER=pi bash install.sh` in a temp environment, count installed kernel files
    - Compare kernel file counts (workflows, references, templates, rules must match)
    - Verify both generate modules.yaml with same module names
    - Use temp HOME dirs to avoid affecting real installation

    **2. Portability Compliance Checks:**
    - Grep all kernel/ files for `@~/` or `@.paul/` patterns (must find zero)
    - Grep all kernel/ files for `~/.pals/` hardcoded paths (must find zero)
    - Grep all kernel/ files for harness-specific tool language: "Use the Read tool", "Use Bash to", "Use Grep to", "Use Edit to", "Use Write to" (must find zero)
    - Verify all `<required_reading>` and `<references>` sections use portable conventions

    **3. Artifact Spec Compliance Checks:**
    - Validate .paul/STATE.md has all 5 required sections from ARTIFACT-SPEC.md
    - Validate .paul/PROJECT.md has required sections
    - Validate .paul/ROADMAP.md has required sections

    **4. Driver Manifest Checks:**
    - Parse each driver.yaml for required capability fields
    - Verify install.sh and uninstall.sh exist for each driver
    - Document capability matrix

    Script outputs results in TAP (Test Anything Protocol) format for easy parsing.
    Script exits 0 if all pass, 1 if any fail.

    Avoid: modifying any existing files, requiring network access, leaving temp files behind (use trap for cleanup).
  </action>
