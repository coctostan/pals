    Add three new subsections to MODULE-SPEC.md section 3 (Hook Composition):

    **3.6 Failure Cascading Across Hook Points (audit #2 + #4)**
    After section 3.5 Data Flow Summary, add a section documenting:
    - When a blocking hook fires (e.g., pre-apply blocks): what happens to remaining modules at that hook point (they don't run — already in 3.2), AND what happens to downstream hook points (post-task, post-apply never fire)
    - The cascade chain: pre-plan block → no plan created, no post-plan; pre-apply block → no tasks execute, no post-task/post-apply; post-task block → remaining tasks skip, post-apply still fires (for cleanup)
    - Recovery guidance: user resolves the blocking reason, then re-runs the phase command
    - Table showing each blocking hook and its downstream impact

    **3.7 Ordering Dependencies (audit #4)**
    Document that:
    - Priority ordering within a hook point is deterministic (already in 3.1)
    - Cross-hook-point ordering is fixed by the kernel loop (pre-plan → post-plan → pre-apply → ...)
    - Modules should NOT assume another module's context_inject is available at the same hook point (only at subsequent hook points)
    - Example: TODD injects tdd_type at pre-plan; WALT cannot read tdd_type at pre-plan, only at post-plan or later

    **3.8 Non-Standard Project Adaptation (audit #5)**
    Add guidance for module detection that finds artifacts but in non-standard form:
    - Three detection outcomes: found-standard, found-non-standard, not-found
    - For found-non-standard: module should inject a warning via context_inject (not block)
    - Example: TODD finds test files but in unconventional paths; DEAN finds package.json but with non-standard dependency format
    - Guidance: warn and adapt, don't silently skip or hard-block

    Keep additions concise — each subsection ~20-30 lines. Total addition ~70-90 lines.
