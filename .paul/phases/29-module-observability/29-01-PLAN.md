<acceptance_criteria>

## AC-1: Per-hook reference mapping
```gherkin
Given a module with hooks registered in module.yaml
When a hook is dispatched by the kernel
Then only the refs listed for that specific hook are loaded (not all module refs)
And each hook entry includes a `refs` array pointing to specific reference files
```

## AC-2: context_inject schema
```gherkin
Given MODULE-SPEC.md documents the context_inject mechanism
When a module developer reads the spec
Then they find a formal schema with required/optional fields, data types, and concrete examples for each hook point
And the flow diagram shows how context_inject data passes between hooks
```

## AC-3: SUMMARY.md frontmatter documented
```gherkin
Given the SUMMARY.md template has frontmatter fields
When a plan completes and SUMMARY.md is generated
Then all fields (requires, provides, affects, tech-stack, key-files, patterns-established) have inline documentation explaining their purpose and expected values
```

## AC-4: Dispatch log output
```gherkin
Given modules are registered for lifecycle hooks
When plan/apply/unify workflows execute hook dispatch steps
Then the workflow outputs a compact dispatch log showing: hook name, modules fired (with priority), and any context_inject keys returned
And the log format is a single line per hook (not verbose)
```

    ```

    ```

    Rules for the dispatch log:
    - One line per hook point (not per module)
    - Show module name and priority in parens

    N. Output dispatch log line:
