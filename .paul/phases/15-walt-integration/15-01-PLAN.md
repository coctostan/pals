<objective>
## Goal
Create a hook-mapping workflow for WALT that maps each kernel lifecycle hook (pre-apply, post-apply, post-unify) to specific reference files and detailed execution instructions — the same pattern TODD uses with plan-phase-tdd.md.

## Purpose
WALT's 6 reference files define rich behavior (quality detection, test running, lint gating, delta tracking, history recording), but the kernel has no explicit instructions for which references to load per hook or what to do with them. The terse hook descriptions in module.yaml ("Capture test baseline") are insufficient for reliable execution. Without this, WALT hooks fire but produce inconsistent results.

## Output
- `modules/walt/workflows/apply-phase-quality.md` — hook-mapping workflow with per-hook sections
- Updated `modules/walt/module.yaml` — register new workflow in files.workflows
- Fixed `modules/walt/workflows/quality-gate.md` — update broken @-references from pre-PALS paths
</objective>

<task type="auto">
  <name>Task 1: Create apply-phase-quality.md hook-mapping workflow</name>
  <files>modules/walt/workflows/apply-phase-quality.md</files>
  <action>
    Create a hook-mapping workflow following the same pattern as modules/todd/workflows/plan-phase-tdd.md.

    Structure:
    - <purpose> section explaining this orchestrates WALT's quality lifecycle
    - <when_to_use> listing the 3 kernel hook dispatch points
    - <references> listing only the WALT reference files
    - <hooks> section with 3 named hooks:

    1. <hook name="pre-apply" priority="100">
       - Title: "Baseline Capture"
       - Load: quality-detection.md, quality-runner.md
       - Action: Detect quality tools via quality-detection.md heuristics. If test runner found, run baseline test capture per quality-runner.md. Store baseline results (passed/failed/skipped counts, coverage %) in context_inject for post-apply comparison.
       - If no tools detected: return context_inject with walt_skip: true. Does NOT block.
       - Return context_inject with baseline data

    2. <hook name="post-apply" priority="100">
       - Title: "Quality Gate"
       - Load: quality-runner.md, quality-lint.md
       - Action: Run result test capture per quality-runner.md, compare to baseline from pre-apply context_inject. Detect regressions per regression detection rules. Then run lint/typecheck/format sequence per quality-lint.md (format first, then lint with auto-fix, then typecheck). Produce combined gate result (most restrictive of test + lint gates).
       - If walt_skip from pre-apply: produce minimal "skipped" reports and continue
       - MAY block if strict mode and regressions/type-errors detected. Offer fix/override/stop.
       - Return annotations with test report + lint report

    3. <hook name="post-unify" priority="100">
       - Title: "Quality History Recording"
       - Load: quality-delta.md, quality-history.md
       - Action: Consume WALT reports from apply annotations. Compute quality delta per quality-delta.md (trajectory indicators, per-metric comparison). Add Quality section to SUMMARY.md. Append entry to .paul/QUALITY-HISTORY.md per quality-history.md (create file if first entry).
       - If no WALT reports (skip): add "Not tracked" to SUMMARY.md quality section. Still append to history with dashes.
       - Does NOT block.

    quality-principles.md is NOT loaded per-hook — it's foundational context, not execution reference.

    Follow the exact XML structure from plan-phase-tdd.md but adapt for WALT's 3 hooks (not 5).
  </action>
  <verify>
    - File exists at modules/walt/workflows/apply-phase-quality.md
    - Contains 3 <hook> sections matching WALT's module.yaml hooks
    - Each hook references specific (not all) reference files
    - Each hook has clear context_inject/annotations return structure
  </verify>
  <done>AC-1, AC-2, AC-3, AC-4 satisfied: hook-mapping workflow maps each hook to specific references with detailed instructions</done>
</task>
