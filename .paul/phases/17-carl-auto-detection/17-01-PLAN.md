<objective>
## Goal
Add codebase auto-detection to CARL so it can scan a project and proactively suggest domain rules based on detected technologies, frameworks, and patterns.

## Purpose
Currently CARL is reactive — users manually create domains and add rules. This makes CARL proactive: it reads the codebase and suggests relevant rules ("I see Next.js — want App Router patterns?"). This is the "CARL gets smarter" brainstorm item.

## Output
- A rule catalog (rule-catalog.yaml) mapping tech signatures to suggested rules
- A scan-codebase task command that detects technologies and suggests domains/rules
- Updated manager.md to expose the `*scan` command
</objective>

  <action>
    Create a YAML catalog that maps technology signatures to suggested rules.

    Structure:
    ```yaml
    technologies:
      nextjs:
        name: "Next.js"
        detect:
          - file: "next.config.*"
          - package_json: "next"
        domain: "nextjs"
        recall: "next,nextjs,app router,server component"
        rules:
          - "Use App Router (app/) over Pages Router (pages/) for new routes"
          - "Prefer Server Components by default — add 'use client' only when needed"
          - "Use next/image for all images (automatic optimization)"
          - "Use next/link for all internal navigation (client-side transitions)"
      react:
        name: "React"
        detect:
          - package_json: "react"
        domain: "react"
        ...
    ```

    Cover at minimum: Next.js, React, Vue, Python (detect pyproject.toml/requirements.txt),
    Go (detect go.mod), Rust (detect Cargo.toml), TypeScript (detect tsconfig.json),
    Node.js (detect package.json without framework), Docker (detect Dockerfile),
    Prisma (detect prisma/schema.prisma), Tailwind CSS (detect tailwind.config.*).

    Each technology should have 3-6 practical rules that enforce idiomatic patterns.
    Rules should be actionable (do X, prefer Y, avoid Z) not vague ("follow best practices").

    Avoid: Making rules too opinionated or framework-version-specific.
    Keep rules at the level of "conventions most teams agree on."
  </action>
  <verify>Read rule-catalog.yaml, confirm valid YAML with at least 8 technology entries, each having detect/domain/rules fields</verify>

<task type="auto">
  <name>Task 2: Create scan-codebase task command</name>
  <files>modules/carl/commands/carl/tasks/scan-codebase.md</files>
  <action>
    Create a new CARL task command that:

    1. Reads rule-catalog.yaml for technology signatures
    2. Scans the current project directory for matching signatures:
       - Check for config files (next.config.*, tsconfig.json, go.mod, etc.)
       - Check package.json dependencies if it exists
       - Check for marker directories (prisma/, .docker/, etc.)
    3. For each detected technology:
       - Check if a domain already exists for it in .carl/
       - If domain exists, check which catalog rules are already present vs missing
       - If domain doesn't exist, flag as "new domain suggestion"
    4. Present results in a clear format:

    ```
    ════════════════════════════════════════
    CARL CODEBASE SCAN
    ════════════════════════════════════════

    Detected Technologies:
      ✓ Next.js (next.config.mjs found)
      ✓ TypeScript (tsconfig.json found)
      ✓ Tailwind CSS (tailwind.config.ts found)
      ✓ Prisma (prisma/schema.prisma found)

    ────────────────────────────────────────
    NEW DOMAIN: nextjs (4 rules)
      1. Use App Router (app/) over Pages Router
      2. Prefer Server Components by default
      3. Use next/image for all images
      4. Use next/link for internal navigation

    NEW DOMAIN: typescript (3 rules)
      1. Enable strict mode in tsconfig.json
      2. Prefer type over interface for object types
      3. Use unknown over any for untyped values

    EXISTING DOMAIN: tailwind (1 new rule)
      + Use @apply sparingly — prefer utility classes in markup

    ────────────────────────────────────────
    [A] Accept all  [S] Select individually  [N] Skip
    ```

    5. On accept: use existing create-domain and add-rule task patterns to create domains/rules
       - For new domains: create domain file + manifest entry with recall keywords
       - For existing domains: append new rules
    6. On select: walk through each domain/rule group for individual accept/reject

    Follow the same markdown command structure as existing tasks (create-domain.md, add-rule.md).
    Reference those tasks for the actual file operations — scan-codebase orchestrates detection
    and presentation, then delegates creation to existing CARL CRUD operations.

    Avoid: Auto-creating rules without user confirmation.
    Avoid: Duplicating CRUD logic — delegate to existing task files.
  </action>
  <verify>Read scan-codebase.md, confirm it has detection logic, presentation format, and delegates to existing CARL tasks for creation</verify>
  <done>AC-2 and AC-3 satisfied: Scan detects technologies and lets user accept/reject suggestions</done>
</task>
