## Goal
Add GitHub Flow enforcement to the three core loop workflows: apply-phase preflight/postflight, unify-phase merge gate, and transition-phase config resolution with PR-first ordering.

## Purpose

The CRITICAL gap is the missing merge gate — UNIFY routes to next PLAN without checking if previous work is merged. This plan closes the critical gap and 3 HIGH-severity gaps by wiring enforcement into the core loop.

## Output
Three workflow files updated with conditional github-flow enforcement that activates when `workflow: "github-flow"` is configured, preserving 100% backward compatibility for legacy/none modes.

## AC-1: Apply-Phase GitHub Flow Preflight
```gherkin
Given pals.json has workflow: "github-flow"
When apply-phase runs after plan approval
Then a github_flow_preflight step validates:
  - Current branch is not base_branch (creates feature branch if needed)
  - Feature branch is not behind base_branch (rebases if update_branch_when_behind)
  - base_branch is up-to-date with remote (fetches origin)
  - Working tree has no unrelated dirty changes
And the step is skipped entirely for legacy/none modes
```

## AC-2: Apply-Phase GitHub Flow Postflight
```gherkin
Given pals.json has workflow: "github-flow"
When apply-phase finishes all tasks successfully
Then a github_flow_postflight step runs:
  - Pushes feature branch if auto_push is true
  - Creates PR via gh pr create if auto_pr is true
  - Records PR URL in STATE.md Git State section
  - Surfaces CI check state if ci_checks is true
And the step is skipped entirely for legacy/none modes
```

## AC-3: Unify-Phase Merge Gate
```gherkin
Given pals.json has workflow: "github-flow" and require_pr_before_next_phase: true
When unify-phase finishes SUMMARY creation (after finalize_summary)
Then a github_flow_merge_gate step checks (in strict order):
  1. PR exists for current branch (create if auto_pr, else prompt)
  2. CI passing (block if ci_checks and failing)
  3. Reviews approved (block if require_reviews and pending)
  4. PR merged (block if not merged, next action = "merge PR")
  5. Local base_branch synced (pull after merge confirmed)
  6. Feature branch deleted (if delete_branch_on_merge)
And routes to specific next-action when any gate fails
And proceeds to check_phase_completion only when all gates pass
And the step is skipped entirely for legacy/none modes
```

## AC-4: Transition-Phase Config Resolution + GitHub Flow Mechanics
```gherkin
Given pals.json may have workflow, branching, or neither
When transition-phase reads git config
Then GIT_WORKFLOW is resolved as: workflow field > branching fallback > "none"
And new fields are read: base_branch, merge_method, delete_branch_on_merge, require_pr_before_next_phase, require_reviews, update_branch_when_behind
And in github-flow mode:
  - commit_phase commits to feature branch (no local merge to base)
  - CI failure in post_commit_automation is BLOCKING (no "merge anyway" escape)
  - merge_method config is passed to gh pr merge
And legacy mode preserves 100% current behavior
```

    ```xml
    <step name="github_flow_preflight" priority="after-approval">
    ```

    Step content:
    - Read workflow mode: `jq -r '.git.workflow // empty' pals.json` with fallback to branching field → "legacy", else "none"
    - If workflow != "github-flow": skip entire step (no-op)
    - Read `base_branch` (default "main"), `update_branch_when_behind` (default true)
    - **Check 1 — Not on base_branch:**
      ```bash
      CURRENT=$(git rev-parse --abbrev-ref HEAD)
      ```
      If on base_branch: create feature branch `feature/{phase-name}` from base_branch
    - **Check 2 — base_branch up-to-date with remote:**
      ```bash
      git fetch origin {base_branch} 2>/dev/null
      LOCAL=$(git rev-parse {base_branch})
      REMOTE=$(git rev-parse origin/{base_branch} 2>/dev/null)
      ```
      If behind: `git checkout {base_branch} && git pull origin {base_branch} && git checkout feature/{phase-name}`
    - **Check 3 — Feature branch not behind base_branch:**
      ```bash
      git rev-list --count {base_branch}..HEAD  # ahead
      git rev-list --count HEAD..{base_branch}  # behind
      ```
      If behind and `update_branch_when_behind`: `git rebase {base_branch}`
      If behind and not `update_branch_when_behind`: warn only
    - **Check 4 — Clean working tree:**
      ```bash
      git status --porcelain
      ```
      If dirty with unrelated changes: surface clearly, ask to stash or commit
    - Display preflight results summary
    - Legacy/none mode: step is entirely skipped

    **2. Add `github_flow_postflight` step** — insert AFTER `post_apply_hooks`, BEFORE `finalize`:

    ```xml
    <step name="github_flow_postflight" priority="after-hooks">
    ```

    Step content:
    - Read workflow mode (same resolution as preflight)
    - If workflow != "github-flow": skip entire step (no-op)
    - Read `auto_push`, `auto_pr`, `ci_checks` from pals.json, plus `remote` and `base_branch`
    - **If auto_push and remote set:**
      ```bash
      git add -A
      git commit -m "feat({phase}): {description}" --allow-empty
      git push origin feature/{phase-name}
      ```
      Display: "Pushed feature/{phase-name} to remote"
    - **If auto_pr and remote set:**
      ```bash
      # Check if PR already exists
      gh pr view feature/{phase-name} --json url 2>/dev/null
      ```
      If no existing PR:
      ```bash
      gh pr create --base {base_branch} --head feature/{phase-name} \
        --title "feat({phase}): {description}" \
        --body "Phase {N} — auto-generated by PALS."
      ```
      Record PR URL in STATE.md `### Git State` section
    - **If ci_checks:**
      ```bash
      gh pr checks feature/{phase-name} --watch
      ```
      Surface result (informational at this stage — blocking happens in merge gate)
    - Legacy/none mode: step is entirely skipped

    **Avoid:**
    - Do NOT modify existing steps — only ADD two new steps
    - Do NOT change legacy/none behavior at all
    - Do NOT make postflight blocking on CI — that's the merge gate's job
    - Keep config resolution DRY: use the same 3-line resolution pattern in both steps

  <name>Task 2: Add GitHub Flow merge gate to unify-phase.md</name>
  <files>kernel/workflows/unify-phase.md</files>
  <action>
    **Add `github_flow_merge_gate` step** — insert AFTER `finalize_summary`, BEFORE `check_phase_completion`:

    ```xml
    <step name="github_flow_merge_gate" priority="after-summary">
    ```

    Step content:
    - Read workflow mode: same resolution pattern (workflow > branching fallback > "none")
    - Read `require_pr_before_next_phase`, `ci_checks`, `require_reviews`, `delete_branch_on_merge`, `base_branch`, `remote`
    - **If workflow != "github-flow" OR require_pr_before_next_phase != true:** skip entirely — route directly to check_phase_completion (preserves legacy/none behavior)

    **Gate checks (strict order — stop at first failure):**

    1. **PR exists:**
       ```bash
       gh pr view feature/{phase-name} --json url,state 2>/dev/null
       ```
       If no PR and `auto_pr`: create via `gh pr create`
       If no PR and not `auto_pr`: prompt user "Create PR for feature/{phase-name}?"
       Record PR URL in STATE.md if newly created

    2. **CI checks passing (if ci_checks=true):**
       ```bash
       gh pr checks feature/{phase-name}
       ```
       If failing → BLOCK. Display:
       ```
       ⛔ MERGE GATE: CI checks failing
       Next action: Fix CI failures, then re-run /paul:unify
       ```
       Do NOT offer "merge anyway". No escape in github-flow mode.

    3. **Reviews approved (if require_reviews=true):**
       ```bash
       gh pr view feature/{phase-name} --json reviewDecision
       ```
       If not approved → BLOCK. Display:
       ```
       ⛔ MERGE GATE: PR review required
       Next action: Get PR reviewed, then re-run /paul:unify
       ```

    4. **PR merged:**
       ```bash
       gh pr view feature/{phase-name} --json state
       ```
       If state != "MERGED" → offer merge:
       ```
       PR is ready to merge. Merge now?
       [1] Yes, merge via gh pr merge | [2] Merge manually, then re-run /paul:unify
       ```
       If "1": read `merge_method` (default "squash"):
       ```bash
       gh pr merge feature/{phase-name} --{merge_method} --delete-branch
       ```

    5. **Local base_branch synced (after merge confirmed):**
       ```bash
       git checkout {base_branch}
       git pull origin {base_branch}
       ```

    6. **Feature branch cleanup (if delete_branch_on_merge):**
       ```bash
       git branch -d feature/{phase-name} 2>/dev/null || true
       ```
       Display cleanup result

    **After all gates pass:**
    - Display:
      ```
      ✓ Merge gate passed — PR merged, base synced, branch cleaned
      ```
    - Update STATE.md Git State: Branch: {base_branch}, PR: merged
    - Proceed to check_phase_completion

    **Next-action routing summary (displayed when gate blocks):**
    ```
    ────────────────────────────────────────
    MERGE GATE STATUS
    ────────────────────────────────────────
    1. PR exists:     ✓/✗
    2. CI passing:    ✓/✗/skipped
    3. Reviews:       ✓/✗/skipped
    4. PR merged:     ✓/✗
    5. Base synced:   ✓/✗
    6. Branch clean:  ✓/✗

    ▶ NEXT: {specific action to unblock}
    ────────────────────────────────────────
    ```

    **Avoid:**
    - Do NOT modify any existing steps
    - Do NOT touch retroactive_unify path — hotfixes are exempt per D6
    - Do NOT add merge gate to route_based_on_completion — the gate step handles all routing
    - Keep the gate as a single new step, not scattered across existing steps
  </action>

    4. CI failure is blocking with no escape in github-flow mode

    **1. Update `read_git_config` step** to add workflow resolution and new fields:

    Add after existing config reads:
    ```bash
    # Workflow mode resolution (workflow > branching fallback > none)
    GIT_WORKFLOW=$(jq -r '.git.workflow // empty' pals.json 2>/dev/null)
    if [ -z "$GIT_WORKFLOW" ]; then
      GIT_BRANCHING_CHECK=$(jq -r '.git.branching // empty' pals.json 2>/dev/null)
      if [ -n "$GIT_BRANCHING_CHECK" ]; then
        GIT_WORKFLOW="legacy"
      else
        GIT_WORKFLOW="none"
      fi
    fi

    # New github-flow fields
    GIT_BASE_BRANCH=$(jq -r '.git.base_branch // "main"' pals.json 2>/dev/null)
    GIT_MERGE_METHOD=$(jq -r '.git.merge_method // "squash"' pals.json 2>/dev/null)
    GIT_DELETE_BRANCH=$(jq -r '.git.delete_branch_on_merge // true' pals.json 2>/dev/null)
    GIT_UPDATE_WHEN_BEHIND=$(jq -r '.git.update_branch_when_behind // true' pals.json 2>/dev/null)
    GIT_REQUIRE_PR=$(jq -r '.git.require_pr_before_next_phase // false' pals.json 2>/dev/null)
    GIT_REQUIRE_REVIEWS=$(jq -r '.git.require_reviews // false' pals.json 2>/dev/null)
    ```

    Note in prose that `GIT_WORKFLOW` drives all conditional behavior below.

    ```
    **If GIT_WORKFLOW = "legacy" or "none":**
    - Current behavior preserved exactly — local merge, hardcoded main references OK
    ```

    Replace hardcoded `main` with `GIT_BASE_BRANCH` in the feature branch merge commands (affects `git checkout main`, `git merge ... into main`, `git fetch origin main`, `git push origin main`). This applies to BOTH github-flow and legacy modes — using the config value is always correct.
