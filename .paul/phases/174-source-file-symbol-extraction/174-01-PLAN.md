<objective>
## Goal
Extend CODI's extractor contract so plans can seed `impact` from repo-relative source files named in PLAN `<context>`, not just from roadmap/objective prose, while preserving v0.1's bounded, no-magic, advisory-only behavior.

## Purpose
Phase 173 proved the real gap is extractor-vs-scope alignment, not codegraph capability. PALS codegraph data is useful on TS-touching scope (`palsHooks` and adjacent Phase 168 symbols resolved with downstream call-sites), but the current extractor only sees prose-heavy markdown tokens and therefore produces CODI_NULL on live planning runs. Phase 174 is the highest-leverage v2.40 iteration: teach the extractor and shared planning workflow to read only the explicitly referenced source files in `<context>`, parse their top-level declarations, and carry those explicit names into the existing planning flow without expanding the PLAN template or stacking new codegraph tools.

## Output
- Updated `modules/codi/module.yaml` so CODI's Step 1 extractor rules explicitly cover PLAN `<context>` source-file lists and bounded top-level declaration extraction for TS/JS source files.
- Updated `modules/codi/references/codi.md` so the Symbol Extraction Rules and call-time contract document the same source-file-driven extraction behavior, ordering, and guardrails.
- Updated `kernel/workflows/plan-phase.md` so `prepare_codi_seed_candidates` performs a bounded read-only pass over explicitly named repo-relative source files, merges those declarations into the existing candidate ordering, and reuses the resulting seeds without changing the PLAN template.
- Updated `tests/pi-end-to-end-validation.sh` and `tests/cross-harness-validation.sh` only if existing validation coverage lacks stable Phase 174 source-file extraction markers.
</objective>
