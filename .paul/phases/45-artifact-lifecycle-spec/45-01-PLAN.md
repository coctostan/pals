<objective>
## Goal
Create three normative specification documents that formalize PALS's portable contract: artifact schemas, lifecycle state machine, and hook dispatch semantics.

## Purpose
These specs are the foundation for all v2.0 portability work. Without them, Phase 46 (skill rewriting) and Phase 47 (pi adapter) would be building against informal conventions. Formalizing the contract ensures Claude Code behavior is preserved (constraint #1) while enabling other harnesses to comply.

## Output
- `docs/ARTIFACT-SPEC.md` — schema for every `.paul/` artifact
- `docs/LIFECYCLE-SPEC.md` — PLAN→APPLY→UNIFY state machine with transitions, preconditions, postconditions
- `docs/HOOK-CONTRACT.md` — hook dispatch semantics, input/output shapes, blocking rules, persistence expectations
</objective>

<task type="auto">
  <name>Task 2: Create LIFECYCLE-SPEC.md</name>
  <files>docs/LIFECYCLE-SPEC.md</files>
  <action>
    Create docs/LIFECYCLE-SPEC.md formalizing the PLAN→APPLY→UNIFY state machine.

    Define the following:

    **Loop States:**
    - READY (no active plan — ready to plan)
    - PLANNING (plan-phase in progress)
    - PLANNED (plan created, awaiting approval)
    - APPLYING (apply-phase executing)
    - APPLIED (apply complete, ready for unify)
    - UNIFYING (unify-phase in progress)
    - UNIFIED (loop closed, ready for next plan or milestone complete)

    **Valid Transitions:**
    - READY → PLANNING (trigger: /paul:plan)
    - PLANNING → PLANNED (trigger: plan-phase completes)
    - PLANNED → APPLYING (trigger: user approves, /paul:apply)
    - APPLYING → APPLIED (trigger: all tasks complete)
    - APPLIED → UNIFYING (trigger: /paul:unify)
    - UNIFYING → UNIFIED (trigger: unify-phase completes)
    - UNIFIED → READY (trigger: transition to next plan or milestone complete)

    **Invalid Transitions:**
    - READY → APPLYING (can't apply without a plan)
    - PLANNED → UNIFYING (can't unify without applying)
    - etc.

    **Per-transition preconditions and postconditions:**
    For each valid transition, define:
    - What artifacts must exist before
    - What artifacts must be created/updated after
    - What STATE.md fields change

    **Phase-level state machine:**
    - Not started → Planning → In progress → Complete
    - How phase state relates to loop state

    **Milestone-level state machine:**
    - Not started → In progress → Complete
    - How milestone state relates to phase completion

    **Fix loop:**
    - Side-loop semantics (standard/fast-forward/hotfix)
    - How fix loops interact with main loop state

    Source all behavior from current workflow files. Do NOT invent new states or transitions — document what exists.
  </action>
  <verify>
    - Walk through a complete PLAN→APPLY→UNIFY cycle and verify each transition matches the spec
    - Verify fix loop states match fix-loop.md workflow
    - Confirm STATE.md field changes are accurate per transition
  </verify>

<task type="auto">
  <name>Task 3: Create HOOK-CONTRACT.md</name>
  <files>docs/HOOK-CONTRACT.md</files>
  <action>
    Create docs/HOOK-CONTRACT.md formalizing hook dispatch semantics.

    Source hook definitions from:
    - kernel/workflows/plan-phase.md (pre-plan, post-plan hooks)
    - kernel/workflows/apply-phase.md (pre-task, post-task, post-apply hooks)
    - kernel/workflows/unify-phase.md (post-unify hook)
    - modules/*/module.yaml (hook registrations)

    For each of the 6 hook points, define:

    **1. pre-plan**
    - Fires: Before scope analysis in plan-phase
    - Input: phase info, project context
    - Outputs: context_inject (type suggestions, candidate files), block (with reason)
    - Persistence: context_inject is ephemeral (prompt-time only)

    **2. post-plan**
    - Fires: After plan created, before state update
    - Input: plan path, plan content, context from pre-plan
    - Outputs: plan_modifications (task restructuring)
    - Persistence: modifications written into PLAN.md

    **3. pre-task**
    - Fires: Before each task in apply-phase
    - Input: task definition, plan context
    - Outputs: context_inject, block
    - Persistence: ephemeral

    **4. post-task**
    - Fires: After each task completes in apply-phase
    - Input: task result, files modified
    - Outputs: context_inject, block (stops remaining tasks but post-apply still fires)
    - Persistence: quality annotations persist to SUMMARY.md

    **5. post-apply**
    - Fires: After all tasks complete (or after post-task block)
    - Input: apply results, all task outcomes
    - Outputs: quality report, context_inject
    - Persistence: quality data persists to WALT history / SUMMARY.md

    **6. post-unify**
    - Fires: After unify-phase reconciliation
    - Input: summary, decisions, deferred issues
    - Outputs: context_inject (advisory only)
    - Persistence: advisory notes in SUMMARY.md

    **General rules:**
    - Priority ordering: ascending (lower number runs first)
    - Blocking: action:block stops workflow, surfaces reason to user
    - Missing modules = no-op (never error on absent module)
    - Each hook loads only its hook-specific refs (not all module refs)

    **Persistence principle:**
    > If a hook result matters to later phases or later harnesses, it must land in an artifact or be derivable from artifacts. Ephemeral prompt-time injection is acceptable only for within-session guidance.

    Source from current module.yaml files and workflow dispatch sections. Do NOT add new hooks or change existing semantics.
  </action>

    - Cross-check against TODD and WALT module.yaml hook registrations
    - Verify persistence expectations match current behavior (WALT writes to quality history, TODD injects at prompt-time)
  </verify>
  <done>AC-3 and AC-4 satisfied: All hooks formalized matching current dispatch behavior</done>
