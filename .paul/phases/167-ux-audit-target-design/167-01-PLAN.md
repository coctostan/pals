<objective>
## Goal
Produce an evidence-backed audit of the current Pi lifecycle surface, define the target UX/design language for the simplified 4-line presentation, and hand Phase 168 an implementation-ready design package.

## Purpose
The user’s complaint is not missing lifecycle capability — it is that the current always-visible Pi lifecycle surface feels noisy, cluttered, and unclear during normal use. Phase 167 should lock the target before code changes begin: what the current widget/status actually render, which parts create noise, what semantic color and hierarchy should communicate, and how far adjacent prompt/notification polish should extend without growing into a new widget system.
