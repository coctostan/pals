<objective>
## Goal
Create the Pi adapter — driver manifest, installer, uninstaller — and wire it into the root installer's driver detection. This gives Pi users a working PALS installation path.

## Purpose
Proves the v2.0 portability story: same kernel, same modules, different harness. Pi becomes the second supported platform after Claude Code.

## Output
- `drivers/pi/driver.yaml` — Pi driver manifest
- `drivers/pi/install.sh` — Pi-specific installer (skill wiring, modules.yaml)
- `drivers/pi/uninstall.sh` — Cleanup script
- Updated root `install.sh` — Pi detection in driver resolution
</objective>

<task type="auto">
  <name>Task 1: Create Pi driver manifest and installer</name>
  <files>drivers/pi/driver.yaml, drivers/pi/install.sh, drivers/pi/uninstall.sh</files>
  <action>
    **driver.yaml:**
    - name: pi
    - platform: pi
    - kernel_version: ">=2.0.0"
    - capabilities: workflow_invoke true, hook_register false (Pi hooks TBD), state_read true, state_write true, user_interact true, command_register false (Pi uses skills, not slash commands)
    - config.framework_dir: "~/.pi/agent/skills/pals"
    - files: [install.sh, uninstall.sh]

    **install.sh:**
    Follow Claude Code installer pattern but target Pi paths:
    1. Validate PALS_ROOT is set
    2. Create ~/.pi/agent/skills/pals/ directory structure (workflows, references, templates, rules)
    3. Copy kernel files from $PALS_ROOT/kernel/ to skill path
    4. Module discovery: read pals.json for enabled modules, parse each module.yaml
    5. Per-module: copy references, workflows, templates, rules, config to skill path
    6. Generate modules.yaml registry at ~/.pi/agent/skills/pals/modules.yaml
    7. Skip: command installation (Pi doesn't use ~/.claude/commands/), hook symlinks, settings.json (Pi has different hook mechanism)
    8. Print summary of installed files

    **uninstall.sh:**
    - Confirm with user before deleting
    - Remove ~/.pi/agent/skills/pals/ directory
    - Print cleanup summary

    Do NOT implement TypeScript extensions or hook dispatch yet — that's for a follow-up plan if needed.
  </action>
  <verify>
    bash -n drivers/pi/install.sh (syntax check)
    bash -n drivers/pi/uninstall.sh (syntax check)
    cat drivers/pi/driver.yaml | python3 -c "import sys,yaml; yaml.safe_load(sys.stdin.read()); print('valid')" 2>/dev/null || echo "check yaml manually"
  </verify>
  <done>AC-1, AC-2, AC-4 satisfied</done>
</task>

<task type="checkpoint:decision" gate="blocking">
  <decision>Pi detection position in root installer</decision>
  <context>
    Root install.sh currently checks: PALS_DRIVER → ~/.claude/ → ANTHROPIC_AGENT_SDK → fallback claude-code.
    Pi detection needs to be inserted. Options differ on where Pi falls in the priority chain, especially if a user has BOTH ~/.claude/ and ~/.pi/.
  </context>
  <options>
    <option id="after-claude">
      <name>Pi after Claude Code (recommended)</name>
      <pros>Claude Code stays default for dual-install users. Pi only auto-detected if no ~/.claude/. Least disruptive.</pros>
      <cons>Users with both harnesses must use PALS_DRIVER=pi explicitly.</cons>
    </option>
    <option id="user-prompt">
      <name>Prompt user when both detected</name>
      <pros>Explicit choice, no hidden behavior. User always knows which driver.</pros>
      <cons>Adds interactive prompt to install flow. More complex logic.</cons>
    </option>
  </options>
  <resume-signal>Select: after-claude or user-prompt</resume-signal>
</task>
