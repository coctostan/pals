<objective>
## Goal
Create the `/paul:fix` command and compressed fix loop workflow, enabling 3 fix modes: standard fix (compressed PLAN-APPLY-UNIFY), fast-forward (minimal overhead for trivial changes), and hotfix (execute first, retroactive UNIFY).

## Purpose
Closes the #1 friction gap identified in competitive analysis: full PLAN-APPLY-UNIFY for 1-file bugs drives users away. Every competitor (Aider, Cline, Devin, SWE-agent, OpenSpec) handles ad-hoc fixes without ceremony. PALS needs pragmatic escape hatches to remain competitive.

## Output
- `kernel/commands/paul/fix.md` — `/paul:fix` command definition (skill entry point)
- `kernel/workflows/fix-loop.md` — compressed fix workflow (all 3 modes)
- `kernel/references/fix-types.md` — fix classification guide (standard/fast-forward/hotfix)
- Modified `kernel/workflows/unify-phase.md` — retroactive UNIFY support for hotfixes
</objective>

## AC-1: Fix Command Entry Point
```gherkin
Given a user encounters a bug or needs a small change
When they run /paul:fix with a description of the issue
Then they are guided through a fix classification (standard/fast-forward/hotfix)
And the appropriate fix mode is invoked
```

## AC-2: Standard Fix Mode (Compressed Loop)
```gherkin
Given a user selects "standard fix" mode
When the fix loop executes
Then a lightweight FIX.md is created (objective + 1-2 tasks + verify, no full AC/boundaries)
And the fix is applied immediately after creation (no separate approval step)
And a compressed SUMMARY is auto-generated on completion
And STATE.md tracks the fix in accumulated context
```

## AC-3: Fast-Forward Mode
```gherkin
Given a user selects "fast-forward" for a trivial change
When the fast-forward executes
Then no FIX.md is created (direct execution)
And the change is made with a git commit message referencing the fix
And a one-line entry is added to STATE.md accumulated context
And the user is NOT forced through PLAN or UNIFY
```

## AC-4: Hotfix Mode (Retroactive UNIFY)
```gherkin
Given a user selects "hotfix" mode (or has already made changes)
When they run /paul:fix --hotfix (or the fix loop detects uncommitted changes)
Then they can execute the fix immediately
And afterward, /paul:unify can be run retroactively to document what was done
And the retroactive UNIFY creates a SUMMARY from git diff + user description
```

## AC-5: Fix Types Reference
```gherkin
Given a developer or module needs to understand fix classification
When they read fix-types.md
Then they can determine which mode fits their situation
And the reference covers: when to use each mode, what gets tracked, what gets skipped
```

  <name>Task 1: Create fix-types reference and /paul:fix command</name>
  <files>kernel/references/fix-types.md, kernel/commands/paul/fix.md</files>
  <action>
    **fix-types.md** — Reference document defining 3 fix modes:

    1. **Standard fix**: For multi-file bugs or changes needing some structure.
       - Creates lightweight FIX.md (objective + tasks + verify — no full AC/boundaries/checkpoints)
       - Auto-approves after creation (no separate /paul:apply step)
       - Auto-generates compressed SUMMARY on completion
       - Still dispatches post-apply and post-unify hooks (WALT quality gate still fires)

    2. **Fast-forward**: For trivial 1-file, obvious changes.
       - NO plan file created
       - Direct execution: describe → fix → commit → log
       - Commits with message: `fix: [description]`
       - Logs one-line entry in STATE.md accumulated context
       - Skips ALL hooks (no module overhead for trivial fixes)
       - Inspired by OpenSpec's `/opsx:ff` command

    3. **Hotfix**: For urgent fixes where you act first, document later.
       - Execute the fix immediately (or detect already-made changes)
       - Run retroactive UNIFY to document what happened
       - Creates SUMMARY from git diff + user-provided description
       - Post-unify hooks still fire (quality history still updated)

    Include a decision matrix:
    | Criteria | Standard | Fast-Forward | Hotfix |
    | Files changed | 2+ | 1 | Any |
    | Urgency | Normal | Low | High |
    | Plan created? | Lightweight FIX.md | No | No (retroactive SUMMARY) |
    | Hooks fire? | post-apply, post-unify | None | post-unify only |
    | Tracked in STATE? | Yes (full) | Yes (one-line) | Yes (full) |

    **fix.md** — Command definition for `/paul:fix`:
    - Accept description as argument or prompt for it
    - Present fix classification with recommendation based on: number of files mentioned, urgency keywords, whether uncommitted changes exist
    - Route to appropriate mode in fix-loop.md workflow
    - If uncommitted changes detected: recommend hotfix mode
    - Format as standard PALS command (objective, execution_context, context, process, success_criteria)
  </action>
  <verify>
    - fix-types.md exists with all 3 modes documented and decision matrix
    - fix.md command exists with proper PALS command structure
    - fix.md references fix-types.md and fix-loop.md
  </verify>

<task type="auto">
  <name>Task 2: Create compressed fix-loop workflow</name>
  <files>kernel/workflows/fix-loop.md</files>
  <action>
    Create the fix-loop workflow implementing all 3 modes:

    **Standard Fix Mode:**
    1. Create lightweight FIX.md in current phase directory:
       - Frontmatter: phase, plan (next available number), type: fix, autonomous: true
       - Sections: objective (from user description), tasks (1-2 max), verify
       - NO acceptance_criteria, boundaries, skills, or checkpoints sections
       - File name: `{phase}-{NN}-FIX.md`
    2. Auto-approve and execute tasks immediately (no separate /paul:apply prompt)
    3. Dispatch post-apply hooks (WALT quality gate)
    4. Auto-generate compressed SUMMARY:
       - What was fixed (from objective)
       - Files changed (from git diff)
       - Verification result
       - File name: `{phase}-{NN}-FIX-SUMMARY.md`
    5. Dispatch post-unify hooks
    6. Update STATE.md accumulated context with fix entry

    **Fast-Forward Mode:**
    1. Confirm description with user (one prompt)
    2. Execute the fix directly (no plan file)
    3. Git commit with `fix: [description]` message
    4. Add one-line entry to STATE.md: `- Fix (ff): [description] — [timestamp]`
    5. Display completion message, return to normal work
    6. NO hook dispatch, NO plan files, NO summary

    **Hotfix Mode:**
    1. If uncommitted changes exist: show git diff, ask user to describe what was fixed
    2. If no changes yet: let user execute the fix, then proceed
    3. Git commit with `hotfix: [description]` message
    4. Flag for retroactive UNIFY (set STATE.md: `Pending retroactive UNIFY`)
    5. When user runs `/paul:unify` next, detect the retroactive flag and generate SUMMARY from git log + description

    **Workflow structure:**
    - Standard PALS workflow format (purpose, when_to_use, loop_context, process steps)
    - Each mode as a separate process branch
    - Common entry: determine mode → branch to mode-specific steps
    - All modes end by updating STATE.md loop position appropriately
  </action>
  <verify>
    - fix-loop.md exists with all 3 mode implementations
    - Standard mode creates FIX.md and auto-executes
    - Fast-forward mode skips plan creation entirely
    - Hotfix mode supports retroactive documentation
    - All modes update STATE.md
  </verify>
  <done>AC-2, AC-3, AC-4 satisfied: all 3 fix modes implemented in workflow</done>
