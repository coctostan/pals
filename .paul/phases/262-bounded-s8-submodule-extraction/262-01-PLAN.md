### Bounded discovery (planning-time, not contract drift)

The contract states: "Import `markActivation` from `./pals-hooks` value-side. This requires promoting `markActivation` from inline closure to top-level export ... The promotion MUST be bounded and reviewable: `markActivation` gains only the `export` keyword and moves out of the `registerPalsLifecycleHooks` closure to a top-level function in `pals-hooks.ts`; its body is unchanged." Plan-time read of `pals-hooks.ts` lines 557, 663â€“691 confirms `markActivation` closes over the `let activationState` declared at line 557 and is the writer for that store; `getActiveActivation` and `consumeActivationTurn` are the readers and the turn-consumer over the same store. Promoting `markActivation` alone with body unchanged is impossible without also moving `activationState` to module scope; once `activationState` is at module scope, `getActiveActivation` and `consumeActivationTurn` MUST also move (otherwise the in-closure copies and the new top-level copy would diverge). This is the same shape as Phase 258's four-symbol `inline â†’ export` promotion (which also moved a closed-over set together: `MAX_VISIBLE_MODULES` + `RECENT_MODULE_ACTIVITY_LOOKBACK` + `parsePalsState` + `collectRecentAssistantTexts`). The bounded discovery is therefore: Phase 262 promotes **four symbols together** (`activationState` `let` to module-private; `markActivation` / `getActiveActivation` / `consumeActivationTurn` to top-level `export function` with bodies unchanged). This satisfies the contract's body-unchanged requirement and is recorded here as a contract-anticipated bounded discovery, not a contract drift.

The Phase 261 contract's exact words anticipate this: "`getActiveActivation` and `consumeActivationTurn` may stay inside `registerPalsLifecycleHooks` if Phase 262 does not need them; if it does, the same `inline â†’ export` discipline applies." Phase 262 implementation needs all three at module scope to keep `markActivation`'s body unchanged, so the same discipline applies to all three.

## Disposition Selection

This plan selects **Disposition A (preferred â€” full sibling extraction)** as the default per the contract. Disposition B (closure factories stay inline) remains the safe fallback if Disposition A surfaces unexpected closure-state coupling during implementation. The Phase 262 SUMMARY MUST record which disposition shipped and why.

Under Disposition A, the new sibling exports three factory functions (`makeRouteCommand`, `makeRouteWrapperCommand`, `makeRegisterQuickActionShortcut`) that accept their dependencies (`pi`, `markActivation`, plus a `carlState.stashedCmdCtx` setter for the registration handler that is **kept byte-for-byte at the call site in `pals-hooks.ts`**, not moved into the factory) explicitly. The registration loop and the five Ctrl+Alt invocation sites still call the factories from `pals-hooks.ts` inside `registerPalsLifecycleHooks`. This produces the largest LOC reduction in `pals-hooks.ts` and the cleanest single-responsibility sibling.

If Disposition A surfaces coupling that cannot be cleanly threaded through the factory's dependency parameter without breaking another Pi extension contract (for example, if the `Key` import or the `carlState.stashedCmdCtx` shape requires non-trivial refactor), implementation MUST stop, fall back to Disposition B (only pure functions + `COMMANDS` + constants + types move), and record the disposition swap in the SUMMARY.
