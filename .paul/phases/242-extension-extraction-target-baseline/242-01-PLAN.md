# Plan 242-01 — Extension Extraction Target Baseline

<objective>

## Goal

Create a source-cited Phase 242 baseline that chooses and bounds multiple candidate Pi extension submodules from `drivers/pi/extensions/pals-hooks.ts`, so Phase 243 can extract one or more approved submodules without reopening installer, lifecycle-authority, or validation-scope decisions.

## Purpose

v2.55 proved one bounded extraction (`module-activity-parsing.ts`) and generalized the Pi extension install surface. v2.56 should now compound that maintainability gain by choosing a safe extraction wave from the remaining `pals-hooks.ts` subsystems. This phase is planning/baseline only: it should make the next extraction slice explicit, support extracting more than one module when safe, and preserve `.paul/*` artifact authority, parent-owned APPLY, command-output validation truth, no hidden state, no telemetry, no auto-approval, no auto-continue, no skipped checkpoints, no UI-only lifecycle decisions, and no inferred merge intent.

## Output

- New doc: `docs/PI-NATIVE-EXTENSION-EXTRACTION-TARGET-BASELINE.md`.
- The doc records the current extraction state, remaining S1-S8 candidate posture, approved Phase 243 wave candidates, required preserved markers, disallowed surfaces, and a rule that Phase 243 may extract one or more approved candidates only when the plan can keep contracts, imports, install behavior, and validation evidence bounded.

</objective>
