<objective>
## Goal
Add git/GH configuration to PAUL's init workflow and pals.json so every project starts with explicit git conventions — branching strategy, GH repo awareness, PR preferences, and push behavior.

## Purpose
Today init-project.md asks zero git questions. All git behavior is ad-hoc. This plan establishes the configuration foundation that Plan 02 will wire into workflows for automated branching, PRs, pushes, and CI checks.

## Output
- `init-project.md` updated with git/GH configuration step
- `git-strategy.md` updated with configurable conventions section
- `pals.json` schema extended with `git` section
- Graceful degradation: works without GH repo (local-only git)
</objective>

## AC-1: Init asks git configuration questions
```gherkin
Given a new project running /paul:init
When the init workflow reaches the git configuration step
Then it asks about: GH repo (yes/URL/no), branching strategy (feature-per-phase/direct-to-main), auto-push preference, and PR preference
And stores answers in pals.json git section
```

## AC-2: pals.json git schema
```gherkin
Given pals.json exists in a project
When the git section is populated
Then it contains: remote (URL or null), branching (feature-per-phase|direct-to-main), auto_push (boolean), auto_pr (boolean), ci_checks (boolean)
And missing git section defaults to safe values (no auto-push, no auto-PR)
```

## AC-3: Graceful degradation without GH
```gherkin
Given a project initialized without a GH repo
When git operations are configured
Then remote is null, auto_pr is false, ci_checks is false
And auto_push is false
And local git operations (commits, branches, tags) still work
```

## AC-4: git-strategy.md references configuration
```gherkin
Given the git-strategy reference document
When a workflow reads it for git conventions
Then it includes a section explaining pals.json git config
And documents default behavior when config is absent
```

<task type="auto">
  <name>Task 2: Add git configuration step to init-project.md</name>
  <files>kernel/workflows/init-project.md</files>
  <action>
    Add a new step "configure_git" between "configure_modules" and "confirm_and_route":

    The step should:
    1. Detect if current directory is a git repo (`git rev-parse --git-dir 2>/dev/null`)
    2. If not a git repo, note: "Not a git repo — git features disabled. Run `git init` to enable later."
    3. If git repo, ask ONE question at a time:

    Question 1 — GH repo:
    ```
    GitHub repository?

    [1] Yes — detect from remote (git remote get-url origin)
    [2] Enter URL manually
    [3] No GitHub repo (local only)
    ```
    - If 1: auto-detect URL from `git remote get-url origin`, confirm with user
    - If 2: prompt for URL
    - If 3: set remote to null

    Question 2 — Branching (only if git repo detected):
    ```
    Branching strategy?

    [1] Feature branch per phase (recommended — creates feature/{phase-name}, merges on complete)
    [2] Direct to main (all commits on main branch)
    ```

    Question 3 — Automation (only if GH repo detected):
    ```
    Git automation? (Enter to accept defaults)

    Auto-push after phase complete: [yes/NO]
    Auto-create PR on phase transition: [yes/NO]
    Wait for CI checks before merge: [yes/NO]
    ```
    - Defaults are NO for all (conservative)
    - Accept "yes" or "y" to enable each

    Store all answers and write to pals.json git section.

    Update the confirm_and_route step to show git config in the summary:
    ```
    Git: {remote or "local only"} | {branching} | push:{yes/no} PR:{yes/no} CI:{yes/no}
    ```
  </action>
  <verify>Read init-project.md, confirm configure_git step exists with all 3 questions and proper conditional logic</verify>
  <done>AC-1, AC-3 satisfied: Init asks git questions, degrades gracefully without GH</done>

    ```xml
    <configuration>
    ## Git Configuration (pals.json)

    Git behavior is configured per-project in `pals.json` under the `git` key.
    Set during `/paul:init` or manually.

    | Field | Values | Default | Effect |
    |-------|--------|---------|--------|
    | remote | URL or null | null | GH repo for PRs and pushes |
    | branching | feature-per-phase, direct-to-main | feature-per-phase | Branch strategy per phase |
    | auto_push | true/false | false | Push after phase/milestone commits |
    | auto_pr | true/false | false | Create PR on phase transition |
    | ci_checks | true/false | false | Wait for CI before merge |

    **When config is absent:** All automation disabled. Commits are local-only.
    Feature branches still created/merged per workflow instructions,
    but no pushes, PRs, or CI checks.

    **Reading config in workflows:**
    ```bash
    # Check if git section exists
    jq -r '.git.remote // empty' pals.json 2>/dev/null
    jq -r '.git.auto_pr // false' pals.json 2>/dev/null
    ```
    </configuration>
    ```

    Keep within 150-line budget. The new section should be ~25 lines.
    Do NOT modify existing sections — only add the new configuration section.
  </action>
