<objective>
## Goal
Wire the pals.json git configuration (established in Plan 31-01) into the three kernel workflows that perform git operations: transition-phase, complete-milestone, and pause-work. Each workflow reads config at its git action point and conditionally enables automation.

## Purpose
Plan 31-01 created the configuration schema; this plan makes it functional. Without this, the git config fields in pals.json are inert — workflows ignore them and behave as hardcoded. After this plan, users who opted into automation during init get auto PRs, auto push, and CI-aware merges.

## Output
- `transition-phase.md` reads git config for branching, auto_pr, auto_push, ci_checks
- `complete-milestone.md` reads git config for auto_push (tag + branch)
- `pause-work.md` reads git config for branching default
</objective>

## AC-1: transition-phase reads git config
```gherkin
Given a project with pals.json git config
When transition-phase runs the commit_phase step
Then it reads branching, auto_pr, auto_push, ci_checks from pals.json
And creates PR via gh if auto_pr=true and remote is set
And pushes after merge if auto_push=true and remote is set
And checks CI status before merge if ci_checks=true
```

## AC-2: transition-phase degrades gracefully
```gherkin
Given a project with no git remote configured (remote=null)
When transition-phase runs the commit_phase step
Then all remote operations are skipped
And local commit and merge proceed normally
And no errors are thrown
```

## AC-3: complete-milestone reads git config
```gherkin
Given a project with auto_push=true and remote set
When complete-milestone runs the git_tag step
Then it pushes the tag and branch to remote automatically
And if auto_push=false, it shows the push command but does not execute
```

## AC-4: pause-work defaults branch from config
```gherkin
Given a project with branching=feature-per-phase in pals.json
When pause-work offers the WIP commit branch choice
Then it defaults to feature branch (option 2)
And if branching=direct-to-main, it defaults to main (option 1)
```

    ```

    </step>
    ```

    Modify the existing `commit_phase` step:
    - In section 1 "Check for feature branches": use GIT_BRANCHING to determine if feature branches are expected
    - If branching=direct-to-main: skip feature branch check/merge, commit directly to main

    Add a new step `post_commit_automation` after `commit_phase`, before `verify_state_consistency`:

    ```
    <step name="post_commit_automation" priority="after-commit">
    **Conditional automation based on pals.json git config.**

    **1. CI Checks (if ci_checks=true and remote set):**
    Before merging feature branch to main:
    ```bash
    # Push feature branch for CI
    git push origin feature/{phase-name}
    # Check CI status
    gh pr checks feature/{phase-name} --watch
    ```
    If CI fails: warn user, offer to merge anyway or wait.
    If no remote: skip CI checks silently.

    **2. Auto PR (if auto_pr=true and remote set):**
    After merge to main (or instead of local merge if PR-based flow preferred):
    ```bash
    gh pr create --base main --head feature/{phase-name} \
      --title "feat({phase}): {phase-description}" \
      --body "Phase {N} complete. Auto-generated by PAUL."
    ```
    Display PR URL. If auto_pr=false: skip silently.

    **3. Auto Push (if auto_push=true and remote set):**
    After commit/merge:
    ```bash
    git push origin main
    ```
    Display: "Pushed to remote: {remote}"
    If auto_push=false: display "Push when ready: git push origin main"

    **When remote is null:** Skip all three silently. Display nothing.
    </step>
    ```

    Avoid: Changing the verify_state_consistency or route_next steps.
    Avoid: Making remote operations mandatory — all must be conditional.
  </action>
