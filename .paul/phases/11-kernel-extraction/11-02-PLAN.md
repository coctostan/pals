<objective>
## Goal
Remove all hardcoded TODD and WALT references from kernel workflows and replace them with generic lifecycle hook dispatch points. When no modules are installed, these hook points are no-ops.

## Purpose
The kernel must be module-agnostic per the Phase 10 architecture. Currently, apply-phase.md has hardcoded WALT steps (walt_baseline, walt_post_apply), unify-phase.md has hardcoded WALT quality delta and TODD TDD audit steps, and plan-phase.md has hardcoded TODD/TDD references. These must become generic hook invocations so the kernel never checks "if TODD installed" — it checks "if hooks registered for this point."

## Output
- kernel/workflows/apply-phase.md — WALT steps → generic pre-apply/post-apply hook dispatch
- kernel/workflows/unify-phase.md — WALT/TODD audit steps → generic pre-unify/post-unify hook dispatch
- kernel/workflows/plan-phase.md — TDD detection/generation refs → generic pre-plan/post-plan hook dispatch
- kernel/templates/SUMMARY.md — WALT/TDD sections → generic "Module Annotations" section
</objective>

    1. Replace the `<references>` section's `@~/.pals/references/tdd-execution.md` line with a comment noting module references are loaded dynamically via hook dispatch
    2. Replace the `walt_baseline` step (lines ~95-106) with a generic `pre_apply_hooks` step that:
       - Reads `~/.pals/modules.yaml` to find modules with `pre-apply` hooks
       - Sorts by priority (ascending)
       - For each registered module: loads its declared reference files, follows its hook description
       - If no modules registered for pre-apply: log "No pre-apply hooks registered — proceeding" and continue
       - Stores any `context_inject` data from hooks for use in later steps
    3. Replace the `walt_post_apply` step (lines ~203-225) with a generic `post_apply_hooks` step using the same pattern
    4. In the `finalize` step: replace WALT report references with generic "Include module annotations from post-apply hooks (if any)" — no hardcoded report format
    5. In `execute_tasks` step: replace TDD-specific conditional (`If plan frontmatter type is "tdd"`) with a generic note that module-provided execution overlays (loaded via post-plan hooks) may modify task execution. Keep the `type: tdd` frontmatter field as a plan attribute but remove the hardcoded reference to `@~/.pals/references/tdd-execution.md`

    **unify-phase.md:**
    1. Replace `<references>` TDD reference with dynamic module loading comment
    2. Replace `audit_tdd_execution` step with a generic `pre_unify_hooks` step that dispatches to all modules registered for pre-unify
    3. Replace `audit_quality_delta` step with inclusion in the generic hook dispatch (WALT's post-unify hook handles this)
    4. Replace `append_quality_history` step similarly — this becomes a post-unify hook responsibility
    5. Keep `audit_skill_invocations` step as-is (this is kernel functionality, not module-specific)

    **plan-phase.md:**
    1. Replace `<references>` TDD references with dynamic module loading comment
    2. In `analyze_scope` step: replace TDD detection heuristics block with a generic `pre_plan_hooks` dispatch that fires before scope analysis. Modules like TODD can inject `context_inject` data (e.g., tdd_candidates, tdd_type suggestion)
    3. In `create_plan` step: replace TDD plan generation reference with a note that post-plan hooks may modify the plan structure. Keep the `type: tdd` frontmatter option but note it's set based on pre-plan hook suggestions rather than kernel logic

    **Hook dispatch pattern (use consistently across all three files):**
    ```
    1. Read ~/.pals/modules.yaml (if exists)
    2. Find modules with hooks registered for [hook-point]
    3. Sort by priority (ascending — lower runs first)
    4. For each module:
       a. Load module's declared reference files as context
       b. Follow module's hook description
       c. Collect context_inject / annotations / modifications
    5. If no modules registered: proceed (no-op, no warning)
    6. If a blocking hook returns block: stop and surface reason
    ```

    **What NOT to do:**
    - Do NOT remove the `type: tdd` frontmatter field from PLAN template — plans can still be typed
    - Do NOT remove the TDD/Quality sections from SUMMARY template (that's Task 2)
    - Do NOT modify any files outside kernel/workflows/
    - Do NOT add conditional `if todd_installed` or `if walt_installed` checks — use generic hook dispatch only
