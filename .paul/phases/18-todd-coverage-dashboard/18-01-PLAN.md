<objective>
## Goal
Add test coverage visibility to PALS via a `/paul:coverage` command that detects the project's test framework, runs coverage analysis, and presents a dashboard mapping tested vs untested code paths.

## Purpose
TODD enforces TDD discipline during plan execution but has no visibility into *where* coverage is thin across the project. This gives users a proactive view of coverage gaps and suggests targeted TDD plans for uncovered areas.

## Output
- `modules/todd/references/coverage-detection.md` — framework-to-coverage-tool mapping and output parsing
- `kernel/commands/paul/coverage.md` — the `/paul:coverage` command
- Updated `modules/todd/module.yaml` — new reference registered
</objective>

    Create a reference document that maps test frameworks to coverage tools and commands.

    Structure:
    1. Detection table — framework indicator file → coverage command
       - package.json with jest → `npx jest --coverage --coverageReporters=text`
       - package.json with vitest → `npx vitest run --coverage`
       - package.json with c8/nyc → `npx c8 npm test` / `npx nyc npm test`
       - pyproject.toml/setup.py → `pytest --cov --cov-report=term`
       - go.mod → `go test -coverprofile=cover.out ./... && go tool cover -func=cover.out`
       - Cargo.toml → `cargo tarpaulin --out stdout` (or `cargo llvm-cov`)
       - mix.exs → `mix test --cover`
    2. Output parsing guidance — how to extract per-file coverage percentages from each tool's text output
    3. Fallback — if no framework detected, ask user for their coverage command

    Follow the style of tdd-execution.md's detect_test_command step.
    Keep it a reference document (not a workflow), using XML sections.
    Do NOT duplicate tdd-execution.md's test command detection — reference it and extend for coverage-specific flags.

<task type="auto">
  <name>Task 2: Create /paul:coverage command</name>
  <files>kernel/commands/paul/coverage.md</files>
  <action>
    Create the command file following the pattern of verify.md (frontmatter with name, description, allowed-tools).

    Frontmatter:
    - name: pals:coverage
    - description: Run test coverage analysis and display dashboard of tested vs untested code
    - allowed-tools: [Read, Bash, Glob, Grep]

    Command workflow:
    1. Detect test framework — follow coverage-detection.md's detection table
       - Check for framework indicator files (package.json, go.mod, etc.)
       - If package.json: read it, check for jest/vitest/c8/nyc in dependencies/devDependencies or scripts
       - Map to coverage command
    2. Confirm and run — show the detected command, run it via Bash
    3. Parse output — extract per-file coverage percentages from the text output
    4. Present dashboard:
       ```
       ════════════════════════════════════════
       TODD COVERAGE DASHBOARD
       ════════════════════════════════════════

       Overall: XX% coverage

       ── Files by Coverage ──────────────────
       [file path]                    [XX%]
       [file path]                    [XX%]
       ...

       ── Untested Files (0%) ────────────────
       [file path]
       [file path]

       ── Summary ────────────────────────────
       Tested: N files | Partial: N files | Untested: N files
       ════════════════════════════════════════
       ```
    5. Suggest actions:
       - If untested files exist: "Consider `/paul:plan` with TDD for [top 3 untested files]"
       - If overall coverage is low: "Run `/paul:coverage src/specific/dir` to focus analysis"
       - If coverage is good: "Coverage looks healthy!"

    Support optional $ARGUMENTS for targeting a specific directory or file.

    Error handling:
    - No framework detected → ask user for coverage command
    - Coverage tool not installed → show install instructions
    - No test files found → "No tests found. Start with `/paul:plan` using type: tdd"

    Reference @modules/todd/references/coverage-detection.md for detection logic.
    Follow scan-codebase.md's pattern: detect → present → suggest actions.
  </action>
