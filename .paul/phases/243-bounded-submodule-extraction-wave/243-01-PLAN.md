# Plan 243-01 — Extract S1 `artifact-slice-rendering` to a Sibling Pi Extension Module

<objective>

## Goal

Extract the S1 `artifact-slice-rendering` subsystem from `drivers/pi/extensions/pals-hooks.ts` into a new sibling Pi extension module `drivers/pi/extensions/artifact-slice-rendering.ts`, keeping `pals-hooks.ts` primarily imports + delegated calls for that subsystem and preserving every visible marker, payload bound, fallback string, authority string, file-source contract, and call-site behavior.

## Purpose

Phase 242 approved S1 as the default primary candidate for the v2.56 Pi extension submodule extraction wave (`docs/PI-NATIVE-EXTENSION-EXTRACTION-TARGET-BASELINE.md`). S1 is the strongest pure/similar pair half: a deterministic build/render chain (`buildArtifactSlice` → `renderArtifactSlices` → `buildPalsContextPayload` → `palsHooks`) over read-only filesystem snapshots that returns derived orientation text. The S5 `module-activity-parsing` extraction (Phase 239) proves the sibling-module pattern; the Phase 241 generalized install surface auto-covers any new repo-owned `drivers/pi/extensions/*.ts` file by basename, so this plan needs no installer, `driver.yaml`, or test-suite edits to deploy. This plan extracts only S1 — S2 `workflow-resource-capsule-rendering` stays inline and is reserved for a later approved Phase 243-02 plan, per the user-approved scoping decision.

## Output

- New repo source file: `drivers/pi/extensions/artifact-slice-rendering.ts` (sibling Pi extension, S5-style header, `Authority: Derived aid only`, single-defined S1 markers, exported chain).
- Updated `drivers/pi/extensions/pals-hooks.ts`: imports `renderArtifactSlices` from `./artifact-slice-rendering`, deletes the inline S1 definitions, removes the now-unused `MAX_ARTIFACT_SLICE_LINES` default-arg from `selectBoundedLines` (which becomes a required parameter), and keeps the existing call site at `renderArtifactSlices(cwd, state)` inside `buildPalsContextPayload` unchanged in behavior.
- Updated `tests/pi-end-to-end-validation.sh`: the three S1 marker `tap_file_contains_all` blocks now point at the new sibling module instead of `$EXT_SRC`, and one new Phase 243-specific TAP guardrail (parallel to the Phase 239 S5 guardrail) asserts the extraction is single-defined and import-wired. Net count moves from `1..212` to `1..213`.
- All four official validation commands pass with command-output evidence; `git diff --check` clean; APPLY also runs the repo installer (`bash drivers/pi/install.sh`) so the generalized install surface deploys both `pals-hooks.ts` and `artifact-slice-rendering.ts` as `[ok] Pi extensions installed: 3 files`.

</objective>
