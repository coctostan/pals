<objective>
## Goal
Create a reusable E2E test protocol and structured report template that codifies the ad-hoc testing approach used in v2.23, v2.25, and v2.26-28 into a repeatable PALS capability.

## Purpose
E2E testing has been the most effective way to find real PALS issues — v2.23 exposed silent module dispatch failure, v2.25 produced 13 actionable items across 3 milestones. But each test required re-specifying everything from scratch. This plan creates a durable protocol that can be loaded during milestone discussion to guide any future E2E test run, and a report template that produces actionable items ready for follow-up milestones.

## Output
- `kernel/references/e2e-test-protocol.md` — the main protocol document
- `kernel/templates/E2E-REPORT.md` — structured report template
</objective>

<task type="auto">
  <name>Task 1: Create E2E Test Protocol Document</name>
  <files>kernel/references/e2e-test-protocol.md</files>
  <action>
    Create the E2E test protocol as a kernel reference document. Structure it as follows:

    **1. Purpose & When to Use**
    - What this protocol is for (structured E2E testing of PALS)
    - When to use it (after major changes, periodic validation, new model evaluation)
    - What it replaces (ad-hoc per-run specification)

    **2. Run Configuration**
    - Project type: greenfield / brownfield (what each exercises)
    - Test posture: adversarial (deliberately test edge cases), real-dev (build something genuine), mixed
    - Model(s) under test: single model deep-dive vs multi-model comparison
    - Focus area: broad audit (all dimensions), targeted (specific modules, workflow steps, or factors)
    - Run scope: full lifecycle (init → multiple loops → milestone complete) vs partial (single loop, specific workflow)

    **3. Monitoring Dimensions Catalog**
    For each dimension, document:
    - What it measures
    - How to assess it (what to look for, what commands/checks to run)
    - Scoring criteria (what constitutes pass/partial/fail)
    - Evidence to collect

    Dimensions (derived from v2.23/v2.25 experience):
    a) **Module Dispatch** — L1 workflow compliance (did the model follow dispatch steps?), L2 actual dispatch (did hooks fire with real commands?), L3 output quality (was the output useful?)
    b) **Artifact Quality** — PLANs (clarity, specificity, AC testability), SUMMARYs (accurate reconciliation, module reports populated), STATE.md (accurate position tracking)
    c) **Workflow Compliance** — Loop integrity (PLAN→APPLY→UNIFY sequence respected), state transitions (STATE.md updated correctly at each step), required steps not skipped
    d) **Git Flow** — Branch creation, PR lifecycle, merge gate execution, CI checks, branch cleanup (only for github-flow projects)
    e) **Init/Onboarding** — Question count, smart defaults, brownfield detection accuracy, artifact completeness (only for greenfield runs)
    f) **Session Management** — CARL boundary detection, handoff quality, resume accuracy, context continuity
    g) **Model Behavior** — Instruction following fidelity, tool usage (runs commands vs describes them), autonomous vs guided behavior, error recovery
    h) **Planning Quality** — Scope sizing, task specificity, checkpoint placement, boundary protection

    Make clear that not all dimensions apply to every run — the run configuration determines which dimensions are active.

    **4. Execution Guide**
    - Pre-run setup: choose project, configure pals.json, set up cmux pane if needed
    - During run: what to observe at each PAUL lifecycle point (init, plan, apply, unify, milestone transitions)
    - Per-phase monitoring: what to check after each loop iteration
    - Real-time notes: capture observations as they happen (don't rely on post-hoc reconstruction)

    **5. Scoring Model**
    - Per-dimension: percentage or pass/partial/fail with evidence
    - Cross-model comparison table format (if multi-model)
    - Overall assessment categories: Critical (blocks value delivery), High (degrades experience), Medium (friction/polish), Low (nice-to-have)
    - How to weight dimensions based on run focus

    **6. Integration with PAUL Lifecycle**
    - How to load this protocol during /paul:discuss-milestone for an E2E test milestone
    - How monitoring dimensions map to phase objectives
    - How the report feeds into the next /paul:discuss-milestone as input

    Use the same markdown conventions as other kernel/references/ files. Keep it practical — this is a working document, not a specification.
  </action>

<task type="auto">
  <name>Task 2: Create E2E Report Template</name>
  <files>kernel/templates/E2E-REPORT.md</files>
  <action>
    Create a structured report template that gets populated during/after an E2E test run. Model it after the v225-actionable-items.md format but make it a reusable template.

    Structure:

    **1. Report Header**
    ```markdown
    # E2E Test Report: {run_name}

    **Date:** {date}
    **Protocol version:** (from e2e-test-protocol.md)
    **Run configuration:**
    | Field | Value |
    |-------|-------|
    | Project | {project_name} ({greenfield/brownfield}) |
    | Model(s) | {model_list} |
    | Posture | {adversarial/real-dev/mixed} |
    | Focus | {broad/targeted: specific_area} |
    | Scope | {full lifecycle / partial: description} |
    | Active dimensions | {list from catalog} |
    ```

    **2. Executive Summary**
    - 3-5 sentence overview of findings
    - Overall health assessment
    - Critical items count

    **3. Dimension Assessments**
    For each active dimension, a standardized section:
    ```markdown
    ## {Dimension Name}

    **Score:** {percentage or pass/partial/fail}
    **Evidence:**
    - {specific observation with artifact/log reference}

    | Check | Result | Notes |
    |-------|--------|-------|
    | {specific check from protocol} | {pass/partial/fail} | {detail} |
    ```

    **4. Cross-Model Comparison** (if multi-model)
    Table format matching v225:
    ```markdown
    |              | L1 Workflow | L2 Dispatch | L3 Quality | ... |
    |--------------|-------------|-------------|------------|-----|
    | {Model A}    | {%}         | {%}         | {%}        |     |
    ```

    **5. Itemized Findings**
    Each finding gets its own section (matching v225 format):
    ```markdown
    ## Item {N}: {Title}

    **Source:** {phase/observation that surfaced this}
    **Problem:** {what's wrong}
    **Evidence:** {specific artifacts, logs, command output}
    **Fix:** {suggested resolution}
    **Risk:** {what could go wrong with the fix}
    **Priority:** {Critical/High/Medium/Low}
    **Milestone candidate:** {suggested future milestone}
    ```

    **6. Milestone Mapping**
    Group findings into suggested milestone candidates:
    ```markdown
    | Milestone | Items | Theme | Priority |
    |-----------|-------|-------|----------|
    | {name}    | {N,M} | {theme} | {Next/After/Defer} |
    ```

    **7. Phase Estimates** (per milestone candidate)
    Rough phase breakdown for each suggested milestone.

    Include template comments (<!-- -->) explaining what goes in each section and when to include/skip optional sections. Make it clear this is filled in during the E2E test, not after.
  </action>
