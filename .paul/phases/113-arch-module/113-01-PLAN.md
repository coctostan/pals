  <name>Task 1: Create ARCH reference files</name>
  <files>modules/arch/references/architecture-patterns.md, modules/arch/references/architecture-drift.md, modules/arch/references/architecture-principles.md</files>
  <action>
    Create 3 reference files following the depth pattern of existing modules (e.g., review-patterns.md):

    **architecture-patterns.md** — Pattern catalog organized by category:
    - Layering patterns: MVC, Clean Architecture, Hexagonal, Onion, N-tier
    - Boundary patterns: module boundaries, API boundaries, service boundaries
    - Convention patterns: file naming, directory structure, import ordering
    - Data flow patterns: unidirectional, event-driven, pub/sub, CQRS
    - Component patterns: composition vs inheritance, dependency injection, facades
    Each pattern should have: name, detection heuristic (what to look for in files), violation signal, and why it matters.

    **architecture-drift.md** — Drift detection heuristics:
    - Layer violations: controller importing from DB directly, UI calling infrastructure
    - Circular dependencies: A imports B imports A
    - Boundary crossings: shared mutable state across modules, direct DB access from API handlers
    - Convention drift: files in wrong directories, naming pattern breaks
    - Structural erosion: god modules (>20 files), orphan files, missing index/barrel files
    Each heuristic should have: what to scan, grep/find command, violation signal, severity.

    **architecture-principles.md** — Core principles:
    - Separation of concerns
    - Dependency inversion (depend on abstractions)
    - Single responsibility at module level
    - Explicit boundaries over implicit coupling
    - Convention over configuration for structure
  </action>

<task type="auto">
  <name>Task 2: Create ARCH module.yaml with A-grade hooks</name>
  <files>modules/arch/module.yaml</files>
  <action>
    Create module.yaml following the established structure.

    Module metadata:
    - name: arch
    - display_name: ARCH
    - description: Architecture & Design Guardian. Detects project structure patterns, audits for layer violations and architectural drift, maps codebase layout for onboarding, and tracks design decisions.
    - version: 1.0.0
    - dependencies: kernel >=0.4.0

    Hooks (A-grade from day one):

    pre-plan (priority 75, advisory):
    - Scan project structure: find . -type f -name '*.ts' -o -name '*.py' -o -name '*.go' -o -name '*.rs' -o -name '*.java' | head -200
    - Detect architecture pattern: look for src/controllers/, src/services/, src/models/ (MVC); src/domain/, src/infrastructure/, src/application/ (Clean/Hexagonal); pages/, components/, lib/ (Next.js/frontend)
    - Map directory tree depth and module boundaries
    - Output architecture summary: '| Layer/Module | Files | Pattern | Notes |'
    - Inject arch_context with: detected_pattern, layer_map, boundary_count, files_per_layer
    - If files_modified cross layer boundaries, warn: 'ARCH: Changes span N layers — review for boundary violations'

    post-apply (priority 225, advisory with blocking on critical violations):
    - For EACH changed file, check layer placement: does a controller/handler file import directly from DB/ORM layer?
    - Scan for circular imports: grep import/require statements, check for A→B→A cycles in changed files
    - Check naming conventions: are new files following the project's established naming pattern?
    - Check for god modules: any directory gaining >20 source files?
    - Output a violation table: '| File | Check | Result |' with PASS/WARN/VIOLATION per check
    - BLOCK if circular dependency introduced — display: 'ARCH BLOCK: Circular dependency detected: {A} → {B} → {A}'
    - WARN on layer violations, naming drift, growing modules

    files.references: architecture-patterns.md, architecture-drift.md, architecture-principles.md
  </action>
  <verify>
    grep -c "BLOCK\|violation table\|layer\|circular\|boundary\|pattern\|architecture" modules/arch/module.yaml
    Expected: 8+ matches
  </verify>
  <done>AC-1 satisfied: module.yaml follows established structure with A-grade hooks.</done>
