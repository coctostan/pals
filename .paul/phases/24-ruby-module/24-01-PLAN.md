<objective>
## Goal
Create the RUBY (Refactor Utility & Better-code Yielder) module as a new first-party pal with a `/paul:refactor` command that scans for tech debt, suggests refactoring opportunities, and provides actionable improvement guidance after milestones.

## Purpose
PALS builds features through plan/apply/unify loops but has no systematic way to address accumulated tech debt. Projects grow cruft silently — duplicated logic, overly complex functions, dead code, naming inconsistencies. RUBY makes code health visible by scanning for refactoring opportunities and providing prioritized, actionable suggestions. This is the 8th pal, completing the full roster before Phase 25 wires all pals into lifecycle hooks.

## Output
- `modules/ruby/module.yaml` — module manifest (no hooks initially, Phase 25 adds hooks)
- `modules/ruby/references/refactor-principles.md` — refactoring philosophy, when to refactor, risk assessment

- `modules/ruby/references/refactor-patterns.md` — common refactoring patterns, safe transformation recipes
- `kernel/commands/paul/refactor.md` — `/paul:refactor` command (scan + suggest + plan modes)

    Create 3 reference files:

    refactor-principles.md:
    - Philosophy: small safe steps, behavior-preserving transformations, always test-backed
    - When to refactor: after milestones (natural pause), before new features (clean foundation), on high-churn files (most value), during bug fixes (understand context already)
    - When NOT to refactor: during active feature work, without tests, under deadline pressure, if it doesn't reduce complexity
    - Risk assessment: blast radius (how many callers affected), test coverage (safe to change?), complexity delta (does refactoring actually simplify?)
    - Boy scout rule: leave code better than you found it, but scope improvements to touched files
    - Refactoring vs rewriting: prefer incremental over big-bang, extract-and-replace over rewrite-in-place

    debt-detection.md:
    - Code smells catalog: duplicated logic, long functions (>50 lines), deep nesting (>3 levels), god objects/files, feature envy, shotgun surgery, dead code, magic numbers/strings
    - Complexity heuristics: cyclomatic complexity thresholds, cognitive complexity, lines per function/file
    - Detection by language: JavaScript/TypeScript (ESLint complexity rules, unused exports), Python (pylint, radon), Go (gocyclo), Rust (clippy complexity)
    - Severity classification: critical (actively causing bugs), high (slowing development), medium (code clarity), low (style/convention)
    - Prioritization: impact × frequency ÷ effort — focus on high-churn, high-complexity files first
    - Structural indicators: file size distribution, import/dependency graphs, circular dependencies

    refactor-patterns.md:
    - Extract function: when a block does one identifiable thing, pull it out with a descriptive name
    - Extract variable: complex expressions → named intermediate variables for readability
    - Inline temp: single-use variables that obscure rather than clarify
    - Replace conditional with polymorphism: switch/if chains on type → interface + implementations
    - Decompose conditional: complex boolean expressions → descriptive helper functions
    - Replace magic values: hardcoded strings/numbers → named constants
    - Consolidate duplicates: repeated blocks → shared function (only when genuinely the same concern)
    - Simplify nesting: early returns, guard clauses, invert conditions to reduce depth
    - Move function: relocate to where its data lives (reduce feature envy)
    - Pattern-to-smell mapping: which pattern addresses which smell
  </action>
