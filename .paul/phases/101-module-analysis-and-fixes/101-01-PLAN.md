<objective>
## Goal
Adversarial testing of TODD, WALT, and DEAN blocking behavior. Deliberately introduce problems through Pi in the test pane and verify the modules actually catch and block them.

## Purpose
Phase 100 proved modules dispatch — but every module said "no issues." We need to verify they actually BLOCK when things go wrong, not just report "all clear."

## Output
- Per-module blocking audit with pass/fail evidence
- Proof that modules catch real problems (or documented failure to catch)
</objective>

<context>
## Test Pane State
- Surface: surface:23, GPT-5.3 Codex, 35.3% context
- pi-verify on main, Phase 4 complete, v0.2 milestone finishing
- All 8 modules dispatching (confirmed in Phase 100 v2)

## What Each Blocking Module Should Do
- TODD post-task: "Run test suite after each task; block if tests regress"
- TODD pre-apply: "Verify test files exist or are first in task order; block if RED phase would be skipped"
- WALT post-apply: "Re-run tests + lint + typecheck; compare against baseline; block if regressions detected"
- DEAN post-apply: "Re-run dep audit; compare against pre-plan baseline; block if new critical/high vulnerabilities"
</context>

<acceptance_criteria>

## AC-1: TODD Catches Test Regression
```gherkin
Given a passing test suite
When a change is made that breaks a test during APPLY
Then TODD's post-task hook detects the regression and blocks
```

## AC-2: WALT Catches Quality Regression
```gherkin
Given a clean lint/typecheck baseline
When code with lint errors or type errors is introduced during APPLY
Then WALT's post-apply hook detects regressions vs baseline and blocks
```

## AC-3: DEAN Catches Vulnerability
```gherkin
Given a clean dependency audit baseline
When a known-vulnerable package is added
Then DEAN's post-apply hook detects the new vulnerability and warns/blocks
```

</acceptance_criteria>
