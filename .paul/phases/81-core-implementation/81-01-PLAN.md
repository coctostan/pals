## AC-1: Command Context Stashing
```gherkin
Given the PALS extension is loaded
When any /paul-* command is invoked
Then the ExtensionCommandContext is stashed in carlState.stashedCmdCtx
And the stashed context is available to event handlers (agent_end, turn_end)
```

## AC-2: Phase Completion Detection
```gherkin
Given a PALS project is active and a /paul-* command was previously invoked
When agent_end fires and STATE.md loop signature has changed to all-complete (✓✓✓)
Then CARL detects the loop-complete state by comparing previousLoopSignature
And CARL evaluates the context usage decision model
```

## AC-3: Session Boundary Decision Model
```gherkin
Given CARL detects a phase completion
When context usage ratio >= continue_threshold (default 0.4) OR strategy is "always-fresh"
Then CARL creates a fresh session with bootstrap context and sends /skill:paul-resume
When context usage ratio < continue_threshold AND strategy is "phase-boundary"
Then CARL notifies user and allows continuation in the same session
```

## AC-4: Mid-Phase Safety Valve
```gherkin
Given a PALS session is active during APPLY
When turn_end fires and context usage ratio >= safety_ceiling (default 0.8)
Then CARL sets pauseAtNextBoundary flag and notifies the user
And the next agent_end triggers a session break regardless of loop state
```

## AC-5: Loop Signature Seeding
```gherkin
Given a PALS session starts (session_start fires)
When STATE.md contains a loop position
Then CARL seeds previousLoopSignature to the current loop state
So that already-complete loops don't trigger spurious session creation on resume
```

    ```typescript
    };

      stashedCmdCtx: any | undefined;
      dispatching: boolean;
      previousLoopSignature: string | undefined;
      pauseAtNextBoundary: boolean;
    };
    ```

    **C. `loadCarlConfig(cwd: string)` helper (add after existing helper functions, before the command definitions):**
    Read `pals.json` from `.paul/pals.json` (first) or `pals.json` in cwd. Parse JSON, extract `modules.carl` section. Return CarlConfig with defaults for any missing fields. Use `readFileOr` for safe file reads.

    **D. `extractLoopSignature(state: PalsStateSnapshot)` helper:**
    Extract the ✓/○ marks from `state.loop` as a compact string (e.g., "✓✓✓" or "✓○○"). Return undefined if no loop data. Reuse the same `matchAll(/[✓○]/g)` pattern from `renderLoopBadge`.

    **E. `carlCreateFreshSession(carlState, ctx, pi, reason)` function:**
    - Guard: if `carlState.stashedCmdCtx` is undefined, notify user and return false
    - Guard: if `carlState.dispatching` is true, return false (reentrancy)
    - Set `carlState.dispatching = true`
    - Build bootstrap content from `parsePalsState(cwd)` — compact summary with milestone, phase, loop, next action, and the reason for the session break
    - Use `Promise.race` with `CARL_NEW_SESSION_TIMEOUT_MS` timeout around `carlState.stashedCmdCtx.newSession({ setup: (sm) => sm.appendMessage({ role: "user", content: [{ type: "text", text: bootstrapContent }], timestamp: Date.now() }) })`
    - On success (not cancelled): call `pi.sendUserMessage("/skill:paul-resume")` for auto-resume
    - Always reset `carlState.dispatching = false` in finally block
    - Return boolean success

    **F. `carlEvaluatePhaseCompletion(carlState, ctx, pi)` function (called from agent_end):**
    - Parse STATE.md via `parsePalsState(cwd)`
    - Extract loop signature
    - If signature is same as `carlState.previousLoopSignature`: return (no change)
    - Update `carlState.previousLoopSignature` to new signature
    - If new signature is NOT all-complete ("✓✓✓"): return (phase not finished)
    - Load CARL config via `loadCarlConfig(cwd)`
    - If strategy is "manual": return (CARL monitoring only)
    - Read `ctx.getContextUsage()`, get `ctx.model?.contextWindow`, compute ratio
    - If strategy is "always-fresh" OR ratio >= continue_threshold:
      - Notify user: `CARL: Phase complete, context at X%. Creating fresh session.`
      - Call `carlCreateFreshSession(carlState, ctx, pi, "phase-complete")`
    - Else (ratio < continue_threshold):
      - Notify user: `CARL: Phase complete, context at X% — continuing in same session.`

    **G. `carlMonitorSafetyCeiling(carlState, ctx)` function (called from turn_end):**
    - Parse STATE.md to check if PALS is active
    - Load CARL config; if strategy is "manual": return
    - If `carlState.pauseAtNextBoundary` is already true: return (already flagged)
    - Read `ctx.getContextUsage()`, compute ratio
    - If ratio >= safety_ceiling:
      - Set `carlState.pauseAtNextBoundary = true`
      - Notify user: `CARL: Context pressure at X%. Will pause at next task boundary.`

    **Implementation constraints:**
    - All CARL functions are module-level (not inside `palsHooks` closure) EXCEPT `carlState` initialization, which goes inside `palsHooks()` to share scope with `pi`
    - Actually, keep all CARL logic inside `palsHooks()` closure per Phase 77 decision — CARL shares `pi`, activation state, and guided-workflow state
    - Use existing `readFileOr`, `parsePalsState`, `compactWhitespace` helpers
    - Bootstrap message format: compact markdown with milestone, phase, loop position, reason, and handoff path
    - Notify via `ctx?.ui?.notify(message, "info")` for informational, `"warning"` for safety ceiling
