## Goal
Adversarially test DEAN (B→A) by introducing a known vulnerable dependency and verifying DEAN detects it and blocks at post-apply. Strengthen hook descriptions if needed based on test results.

- Adversarial test results showing DEAN blocking on a known vulnerability
- Updated hook descriptions if gaps found during testing
- Reinstall if descriptions change

## AC-1: DEAN pre-plan detects vulnerabilities
```gherkin
Given a project with known vulnerable dependencies
When DEAN pre-plan hook dispatches
Then it outputs a severity breakdown with critical/high/moderate/low counts
And injects dep_warnings into plan context
```

## AC-2: DEAN post-apply blocks on new critical/high vulnerabilities
```gherkin
Given a pre-plan baseline and a newly introduced vulnerable dependency
When DEAN post-apply hook dispatches
Then it detects the NEW vulnerability by comparing against baseline
And it BLOCKs execution with specific package name and severity
```

## AC-3: DEAN at A grade on adversarial retest
```gherkin
Given the adversarial test scenario
When DEAN hook descriptions are evaluated
Then DEAN produces specific, actionable output (not just "27 advisories")
And DEAN correctly differentiates new vs pre-existing vulnerabilities
And DEAN blocks on critical/high, warns on moderate, ignores low
```

<task type="auto">
  <name>Task 1: Strengthen DEAN hook descriptions for adversarial-grade output</name>
  <files>modules/dean/module.yaml</files>
  <action>
    Review current DEAN hook descriptions against A-grade criteria:
    
    For pre-plan hook, ensure description instructs to:
    1. Detect package manager (look for package.json/pnpm-lock.yaml/yarn.lock/go.mod/Cargo.toml/requirements.txt)
    2. Run the appropriate audit command from vulnerability-audit.md
    3. Parse JSON output for severity counts
    4. Output a severity table: | Severity | Count | with critical/high/moderate/low rows
    5. Record baseline counts for post-apply comparison
    6. If audit tool not available, show install command and skip gracefully
    
    For post-apply hook, ensure description instructs to:
    1. Re-run the same audit command
    2. Compare severity counts against pre-plan baseline
    3. Compute DELTA: new critical, new high, new moderate, new low
    4. BLOCK if new_critical > 0 OR new_high > 0 — display specific package names and CVE IDs
    5. WARN (non-blocking) if new_moderate > 0
    6. IGNORE new_low
    7. Output comparison table: | Severity | Before | After | New | with blocking status per row
    
    Also ensure the pre-plan hook description includes "block" in its text since DEAN is an enforcement module — it can block plan creation if critical vulnerabilities exist at planning time.
  </action>
  <verify>
    grep -c "BLOCK\|baseline\|severity table\|package name\|CVE\|comparison table\|new_critical\|new_high" modules/dean/module.yaml
    Expected: 6+ matches confirming all A-grade criteria present
  </verify>
  <done>AC-1 and AC-2 criteria embedded in hook descriptions. AC-3 structural requirements met.</done>
</task>
