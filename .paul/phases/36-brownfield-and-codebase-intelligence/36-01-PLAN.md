## Research Context (from Phase 36 planning research)
**OpenSpec brownfield approach:**
- "Reverse-generate specs" — AI reads existing code, produces markdown specs documenting current behavior
- Three-layer context: project.md (global), AGENTS.md (behavioral), tasks.md (per-change)
- Change isolation in separate directories prevents "catastrophic forgetting"
- Init is minimal scaffold; brownfield intelligence is AI-directed, not automated tooling

**Aider repo map approach:**
- tree-sitter extracts tags (definitions + references) → NetworkX graph → PageRank ranking
- Weight multipliers: chat-mentioned identifiers x10, private symbols x0.1, generic (>5 files) x0.1
- PersonalizationVector: files in active context get 100x weight
- Output: ranked file list with key symbols, fitted to token budget (default 1024 tokens)
- Core insight: the value is the weighted graph + PageRank, not tree-sitter itself

**What PALS adopts (without tree-sitter):**
- Reference graph from @-mentions, file paths, cross-references in markdown + source imports
- Relevance ranking by: reference count, recency, centrality (hub files)
- Token-budget-aware output for context loading
- Brownfield init = detect existing code → map-codebase → reverse-generate PROJECT.md

## AC-1: Brownfield Detection in Init
```gherkin
Given an existing codebase with source files (package.json, src/, etc.)
When user runs /paul:init
Then init detects existing code and asks "Is this an existing project?"
And if yes, routes to /paul:map-codebase before creating PROJECT.md
And uses codebase map findings to pre-populate PROJECT.md fields
```

## AC-2: Map-Codebase v2 Reference Graph
```gherkin
Given an existing codebase
When user runs /paul:map-codebase
Then the workflow produces a GRAPH.md alongside the existing 7 documents
And GRAPH.md contains: file nodes, cross-reference edges, hub files (most referenced), entry points
And hub files are ranked by reference count (most connected first)
```

## AC-3: Map-Codebase v2 Brownfield Context
```gherkin
Given a brownfield codebase mapped with /paul:map-codebase
When the map completes
Then CONVENTIONS.md and ARCHITECTURE.md contain enough detail for Claude to follow existing patterns
And a "Brownfield Quick Start" section in map output tells the user what to do next
And the command offers to run /paul:init after mapping (if .paul/ doesn't exist yet)
```
