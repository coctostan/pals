<objective>
## Goal
Eliminate all hardcoded `feature/${PHASE_NAME}` branch name assumptions from GitHub Flow workflow steps. Fix pause-work staging to capture all repo changes instead of only `.paul/ src/`.

## Purpose
GitHub Flow correctness requires operating on the actual current branch, not an inferred name. Users may be on manually named branches, branches not derived from phase names, or short-lived branches representing one mergeable unit. Similarly, WIP commits during pause must capture all changed files, not just a hardcoded subset.

## Output
4 updated kernel workflow files with branch-name-agnostic GitHub Flow steps and comprehensive staging.
</objective>

## AC-1: Branch name resolved from git, not inferred
```gherkin
Given a GitHub Flow workflow step that operates on the current branch
When the step executes
Then it uses `CURRENT_BRANCH=$(git rev-parse --abbrev-ref HEAD)` or equivalent — never `feature/${PHASE_NAME}` after initial branch detection
```

## AC-2: Preflight may create default branch but downstream uses detected name
```gherkin
Given the apply-phase preflight creates a new branch because user is on base_branch
When subsequent preflight, postflight, and merge gate steps reference the branch
Then they re-detect via `git rev-parse --abbrev-ref HEAD` rather than assuming the created name
```

## AC-3: PR operations use actual branch
```gherkin
Given a workflow step that creates, views, checks, or merges a PR
When the step invokes `gh pr create/view/checks/merge`
Then it uses the detected CURRENT_BRANCH, not a constructed `feature/${PHASE_NAME}` string
```

## AC-4: Pause staging captures all repo changes
```gherkin
Given a PALS project with changes in any directory (kernel/, modules/, drivers/, docs/, root files)
When the user pauses work and commits WIP
Then `git add -A` (or equivalent comprehensive staging) is used — not `git add .paul/ src/`
```
