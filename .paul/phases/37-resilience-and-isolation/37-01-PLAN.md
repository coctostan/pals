<objective>
## Goal
Add optional git worktree isolation during APPLY phase and AGENTS.md generation, completing the v1.0 Resilience & Isolation phase.

## Purpose
Worktree isolation prevents polluting main/feature branches during APPLY — if execution fails or produces bad output, the worktree is discarded with zero cleanup. This is the approach that drove Superpowers to 27K stars. Making it **optional** (config-driven, default off) respects users who prefer direct commits.

AGENTS.md is an emerging convention for machine-readable project instructions that improves interoperability with other AI tools.

  <files>kernel/workflows/apply-phase.md, kernel/workflows/init-project.md, kernel/workflows/transition-phase.md, kernel/references/worktree-isolation.md</files>
  <action>
    **1. Create kernel/references/worktree-isolation.md** (~80 lines)
    Reference doc explaining:
    - What worktree isolation is (git worktree create for APPLY, merge back on success)
    - When to use it (multi-phase projects, risky changes, team environments)
    - How it works in PALS (create before tasks, merge on transition, discard on failure)
    - Config: pals.json `git.worktree_isolation: true|false` (default: false)
    - Commands: `git worktree add`, `git worktree remove`, `git worktree list`
    - Worktree naming: `.worktree-{phase-name}` directory at project root
    - Branch naming: `worktree/{phase-name}` (distinct from `feature/` branches)

    **2. Add worktree step to apply-phase.md**
    Insert a new step `setup_worktree` between `verify_required_skills` and `pre_apply_hooks`:
    - Read pals.json `git.worktree_isolation` config
    - If false or absent: skip (no-op, existing behavior)
    - If true:
      ```bash
      git worktree add .worktree-{phase-name} -b worktree/{phase-name}
      cd .worktree-{phase-name}
      ```
    - Record worktree path in STATE.md under Session Continuity
    - All subsequent task execution happens in worktree directory
    - Add cleanup note: if APPLY fails fatally, offer to discard worktree

    **3. Add worktree merge-back to transition-phase.md**
    In the git_operations step, add a worktree check BEFORE the existing feature branch logic:
    - Check: `git worktree list` for active worktree matching current phase
    - If worktree found:
      - Merge worktree branch into source branch (same logic as feature branch merge)
      - Remove worktree: `git worktree remove .worktree-{phase-name}`
      - Delete worktree branch: `git branch -d worktree/{phase-name}`
    - If no worktree: fall through to existing feature branch logic (unchanged)

    **4. Add worktree question to init-project.md**
    In the `configure_git` step, after branching strategy question, add:
    ```
    Question 3b — Worktree Isolation:

    Use git worktrees during APPLY phase?
    This creates an isolated copy for each phase — if something goes wrong,
    discard the worktree instead of reverting commits.

    [1] No (default) — work directly on current branch
    [2] Yes — isolate each APPLY phase in a worktree
    ```
    - If "1" or enter: `git_worktree_isolation = false`
    - If "2": `git_worktree_isolation = true`
    - Add to pals.json template: `"worktree_isolation": false` in git section

    **Avoid:**
    - Do NOT make worktree the default — must be opt-in
    - Do NOT change behavior when worktree_isolation is false or missing
    - Do NOT add worktree logic to fix-loop workflows (fix-loop is lightweight by design)
    - Keep worktree reference under 100 lines (context budget)
  </action>
