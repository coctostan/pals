    Update each module's `hooks: {}` section with appropriate lifecycle hooks.
    Follow the pattern established by TODD and WALT: each hook has priority and description.

    **Priority scheme** (lower = runs first at each hook point):
    - TODD: pre-plan(100), post-plan(100), pre-apply(50), post-task(100), post-apply(200)
    - WALT: pre-apply(100), post-apply(100), post-unify(100)
    - New modules should use priorities that interleave sensibly:

    **DEAN (Dependency Evaluation & Audit Notifier):**
    ```yaml
    hooks:
      pre-plan:
        priority: 50
        description: Check for outdated/vulnerable dependencies; inject warnings into plan context
      post-apply:
        priority: 150
        description: Verify no new vulnerable dependencies were introduced by changes
    ```
    Rationale: pre-plan at 50 (before TODD's 100) — dependency issues should surface before TDD detection. post-apply at 150 (between WALT's 100 and TODD's 200) — after quality gate, before refactor signal.

    **IRIS (Intelligent Review & Inspection System):**
    ```yaml
    hooks:
      pre-plan:
        priority: 150
        description: Scan planned files for known anti-patterns; inject review context
      post-apply:
        priority: 250
        description: Run review patterns against changed files; report findings
    ```
    Rationale: pre-plan at 150 (after TODD's 100) — review context after TDD detection. post-apply at 250 (after TODD's 200) — review runs last after quality and refactor signals.

    **SKIP (Smart Knowledge & Information Persistence):**
    ```yaml
    hooks:
      post-apply:
        priority: 300
        description: Identify decisions made during apply for potential knowledge capture
      post-unify:
        priority: 200
        description: Extract key decisions from SUMMARY.md; suggest knowledge entries
    ```
    Rationale: post-apply at 300 (runs last) — knowledge capture is observational, after all other checks. post-unify at 200 (after WALT's 100) — capture knowledge after quality delta recorded.

    **DAVE (Deploy Automation & Verification Engine):**
    ```yaml
    hooks:
      pre-plan:
        priority: 200
        description: Check if planned changes affect CI/CD configs; flag for review
      post-apply:
        priority: 175
        description: Validate any modified pipeline configs remain syntactically valid
    ```
    Rationale: pre-plan at 200 (after IRIS's 150) — CI/CD context is supplementary. post-apply at 175 (between DEAN's 150 and TODD's 200) — pipeline validation before refactor signal.

    **RUBY (Refactor Utility & Better-code Yielder):**
    ```yaml
    hooks:
      pre-plan:
        priority: 250
        description: Check if high-debt files are being touched; inject refactoring context
      post-unify:
        priority: 300
        description: Scan changed files for new tech debt introduced; suggest improvements
    ```
    Rationale: pre-plan at 250 (runs last) — refactoring context is lowest priority input. post-unify at 300 (after SKIP's 200) — debt assessment is the final observation.

    **Important:**
