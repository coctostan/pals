<purpose>
TDD execution overlay for apply-phase.md. Invoked when the plan being applied has `type: tdd` in its frontmatter.
Enforces RED-GREEN-REFACTOR sequencing with phase-gating and atomic commits per phase.

**This workflow is invoked by apply-phase.md when plan type is tdd.**
It does not replace apply-phase.md — it provides TDD-specific execution logic for the tasks.
</purpose>

<when_to_use>
- apply-phase.md is executing a plan with `type: tdd` in frontmatter
- Plan contains RED/GREEN/REFACTOR task structure (generated by plan-phase-tdd.md)
</when_to_use>

<references>
@~/.pals/references/tdd.md
@~/.pals/workflows/plan-phase-tdd.md
</references>

<process>

<step name="detect_test_command" priority="first">
Before executing any phase, determine the project's test command:

1. Check for common test runners in order:
   - `package.json` → `npm test` or `npx jest` or `npx vitest`
   - `Cargo.toml` → `cargo test`
   - `pyproject.toml` or `setup.py` → `pytest`
   - `go.mod` → `go test ./...`
   - `mix.exs` → `mix test`
   - `Makefile` with test target → `make test`
2. If no test runner detected: ask user for test command
3. Store as `test_command` for use in all phases
4. Optionally store `test_command_focused` for running specific test file (e.g., `npx jest {file}`)
</step>

<step name="execute_red" gate="entry">
**RED PHASE: Write failing test**

1. Read the RED task from the TDD plan (first task, named "RED: ...")
2. Read the `<feature><behavior>` section for test case specifications
3. Create test file following project conventions:
   - Use project's test file naming pattern
   - Import/reference the source module (even though it doesn't exist yet)
   - Write tests covering each behavior case from the plan
   - Use descriptive names: "should [behavior] when [condition]"
4. Run `test_command`:

   **Expected: Tests FAIL**
   - Log: `✓ RED verified — tests fail as expected`
   - Verify failure is for the RIGHT reason:
     - Missing module/function → correct (implementation doesn't exist yet)
     - Syntax error in test → STOP: "Test has syntax errors. Fix before proceeding."
     - Import error for test framework → STOP: "Test framework not configured. Set up before proceeding."

   **Unexpected: Tests PASS**
   - STOP execution immediately
   - Display:
     ```
     ════════════════════════════════════════
     ⚠️  RED PHASE FAILED: Tests pass unexpectedly
     ════════════════════════════════════════

     Tests were expected to FAIL but they PASS.
     This means either:
       1. The feature already exists
       2. The test isn't testing what you think

     Investigate before proceeding.

     [1] I'll fix the test | [2] Feature exists, skip TDD | [3] Stop
     ════════════════════════════════════════
     ```
   - Wait for user input before proceeding

5. Create atomic commit:
   ```
   git add [test file]
   git commit -m "test({phase}-{plan}): add failing test for [feature]

   - [test case 1 description]
   - [test case 2 description]
   - [edge case description]"
   ```
6. Store `red_commit_hash` for UNIFY audit
7. Mark RED as complete, unlock GREEN
</step>

<step name="execute_green" gate="red_complete">
**GREEN PHASE: Implement to pass tests**

**GATE: RED must be complete.** Check that `red_commit_hash` exists.
If not: STOP — "Cannot start GREEN without verified RED phase."

1. Read the GREEN task from the TDD plan (second task, named "GREEN: ...")
2. Read the `<feature><implementation>` section for approach guidance
3. Write MINIMAL implementation:
   - Only what's needed to pass the tests
   - No cleverness, no optimization
   - Follow the test's expectations exactly
4. Run `test_command`:

   **Expected: ALL tests PASS (new and existing)**
   - Log: `✓ GREEN verified — all tests pass`

   **If new tests fail:**
   - Iterate on implementation (DO NOT change tests)
   - Re-run test_command
   - Maximum 3 iterations before stopping:
     ```
     ════════════════════════════════════════
     ⚠️  GREEN PHASE STALLED: Tests still failing after 3 attempts
     ════════════════════════════════════════

     Failing tests:
     [test output summary]

     [1] Continue iterating | [2] Review test expectations | [3] Stop
     ════════════════════════════════════════
     ```

   **If existing tests break:**
   - Fix implementation without changing new test expectations
   - Re-run to verify all tests pass

5. Create atomic commit:
   ```
   git add [source file(s)]
   git commit -m "feat({phase}-{plan}): implement [feature]

   - [brief implementation approach]"
   ```
6. Store `green_commit_hash` for UNIFY audit
7. Mark GREEN as complete, unlock REFACTOR
</step>

<step name="execute_refactor" gate="green_complete">
**REFACTOR PHASE: Clean up (optional)**

**GATE: GREEN must be complete.** Check that `green_commit_hash` exists.
If not: STOP — "Cannot start REFACTOR without verified GREEN phase."

1. Read the REFACTOR task from the TDD plan (third task, named "REFACTOR: ...")
2. Review implementation for obvious improvements:
   - Extract constants for magic values
   - Rename for clarity
   - Remove duplication
   - Simplify logic
3. If NO improvements needed:
   - Log: `REFACTOR skipped — no improvements needed`
   - Mark as complete (no commit)
   - Proceed to return_results

4. If improvements made:
   a. Apply changes
   b. Run `test_command`:

      **Expected: ALL tests STILL PASS**
      - Log: `✓ REFACTOR verified — tests still pass`

      **If any test breaks:**
      - UNDO the specific change that broke it
      - Re-run tests to confirm restoration
      - Log: `REFACTOR partially applied — reverted change that broke tests`

   c. Create atomic commit (only if changes remain):
      ```
      git add [modified file(s)]
      git commit -m "refactor({phase}-{plan}): clean up [feature]

      - [what was improved]
      - No behavior changes"
      ```
   d. Store `refactor_commit_hash` for UNIFY audit
</step>

<step name="return_results">
Return execution results to apply-phase.md for its finalize step:

```
TDD Execution Results:
  RED:      ✓ verified (commit: {red_commit_hash})
  GREEN:    ✓ verified (commit: {green_commit_hash})
  REFACTOR: ✓ complete / skipped (commit: {refactor_commit_hash} / none)

  Total commits: 2-3
  All tests passing: yes
```

These results feed into apply-phase.md's finalize step and then into UNIFY.
</step>

</process>

<output>
- Test file created and committed (RED)
- Source file created/modified and committed (GREEN)
- Optional cleanup committed (REFACTOR)
- 2-3 atomic commits following tdd.md pattern
- Execution results for apply-phase.md finalize
</output>

<error_handling>
**Test command not found:**
- Ask user: "What command runs your tests?"
- Store response as test_command

**RED passes unexpectedly:**
- Stop and present investigation options
- Do NOT proceed to GREEN — this undermines TDD's purpose

**GREEN fails after 3 iterations:**
- Stop and present options
- Likely indicates test expectations or implementation approach needs rethinking

**REFACTOR breaks tests:**
- Undo the breaking change immediately
- Either skip refactor or try smaller changes
- Never commit code that breaks tests

**Git commit fails:**
- Check if git is initialized
- Check for pre-commit hooks that may block
- Report to user with options
</error_handling>

<anti_patterns>
**Changing tests to match implementation:**
Tests define the contract. Implementation conforms to tests, not the other way around.
If tests seem wrong, go back to RED and fix them there.

**Skipping RED verification:**
If you don't verify the test fails first, you don't know it's testing the right thing.

**Over-engineering in GREEN:**
Write the simplest thing that passes. Elegance comes in REFACTOR.

**Adding new behavior in REFACTOR:**
REFACTOR changes HOW, not WHAT. All behavior changes happen in RED (new test) + GREEN (implementation).

**Batching commits:**
Each phase gets its own commit. This creates an auditable trail for UNIFY.
</anti_patterns>
